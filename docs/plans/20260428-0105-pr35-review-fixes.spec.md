# Spec: PR 35 Review Fixes

## Summary

PR #35 replaces the listr2 integration with native runner behavior. The remaining inline review comments identify five regressions in parent task propagation, dynamic subtask paths, prompt renderer erase semantics, and terminal line wrapping width. This work addresses those review items without expanding the feature scope of the branch.

## Goals

- Preserve parent/child task semantics when `createListr` receives a parent task.
- Ensure dynamically added tasks report the same parent-prefixed paths as constructor-created child tasks.
- Match terminal CSI erase behavior for `CSI 1 K` and `CSI 1 J`.
- Wrap terminal output by display columns for non-ASCII text while preserving terminal control sequences.
- Add focused regression tests for each accepted review item.
- Reply to every actionable inline review thread with the outcome and commit the applied fixes.

## Non-Goals

- Redesign the task runner API beyond optional parent propagation.
- Replace the renderer's terminal emulator with a full ANSI parser.
- Add new runtime dependencies solely for display-width handling.
- Address non-actionable acknowledgement replies from the review bot.
- Modify documentation outside the required planning records unless verification reveals a user-facing behavior document is stale.

## Current State

- `packages/core/src/utils/listr.ts` accepts a third `parentTask` argument in `createListr` but does not pass it to the runner.
- `packages/runner/src/executor.ts` prefixes constructor-created child task paths with `parentTask.path`, but `PubmTaskRunner.add()` creates paths from only the dynamic task title.
- `packages/runner/src/renderer.ts` implements `eraseLine(1)` without erasing the cursor cell, and `eraseScreen(1)` rewrites rows while resetting `row` and `column` to zero.
- `packages/runner/src/text.ts` uses stripped string length as the wrapped column count, which is wrong for wide non-ASCII characters.
- Existing tests cover nested `newListr` paths, root `add()` paths, prompt capture control handling, and ANSI/control wrapping, but not the five review regressions.
- `@pubm/runner` only declares `@clack/prompts` as a dependency, so transitive display-width packages must not be imported directly.

## Audience And Use Cases

- pubm CLI users who rely on accurate task nesting, rendering, and prompt output in interactive and CI modes.
- SDK consumers using the runner compatibility surface through `createListr`, `newListr`, or dynamic task addition.
- Reviewers validating that PR #35 preserves listr-style behavior while moving to native runner primitives.

## Requirements

- `createListr(task, options, parentTask)` must create a subtask runner associated with `parentTask` when that argument is supplied.
- `createTaskRunner` may gain an optional parent argument if needed, but existing two-argument callers must remain compatible.
- `PubmTaskRunner.add()` must create dynamic `RuntimeTask` paths using `parentTask.path` for child runners and single-title paths for root runners.
- `eraseLine(1)` must blank from the start of the current line through the cursor cell without moving the cursor.
- `eraseScreen(1)` must blank from screen start through the cursor cell without changing the cursor position.
- `wrapTerminalLine()` must count visible display columns for non-control characters and keep terminal controls attached without counting them as width.
- Regression tests must fail on the current behavior and pass after the implementation.
- Review thread replies must begin with `@cubic-dev-ai` and be posted as inline thread replies.

## Interfaces And Contracts

- Public runner factory: `createTaskRunner<Context>(task, options?, parentTask?)`.
- Core compatibility wrapper: `createListr<Context>(task, options?, parentTask?)`.
- Task paths in `RuntimeTaskSnapshot.path` and `task.subtasks` events must stay stable for root tasks and become parent-prefixed for child dynamic tasks.
- Prompt output capture must preserve cursor row and column for `CSI 1 J`; full-screen erase mode `CSI 2 J` may continue resetting the cursor.
- Terminal line wrapping remains a string-array API and must preserve original escape/control bytes across wrapped chunks.

## Constraints

- Keep changes narrowly scoped to review-fix files and tests.
- Do not revert unrelated work or commit unrelated changes.
- Use existing project style, Bun/Vitest commands, and TypeScript strictness.
- Avoid dependency churn for the local display-width fix.
- Preserve existing ASCII source style unless existing test strings require Unicode to verify width behavior.

## Acceptance Criteria

- Focused runner tests prove `createListr` parent propagation and child-runner `add()` path propagation.
- Focused renderer tests prove `CSI 1 K` erases the cursor cell and `CSI 1 J` leaves cursor position intact for subsequent writes.
- Focused text tests prove a wide non-ASCII string wraps by terminal display columns and still preserves control sequences.
- `bun vitest --run` for the affected packages or narrower package-level test files passes.
- Type checking or the smallest meaningful TypeScript verification for affected packages passes.
- Every actionable review thread receives a factual reply with reviewer mention.
- A commit contains only the applied P2 review fixes and their planning/test support.

## Risks

- Display-width handling can be incomplete for every Unicode edge case; the local helper must cover common combining marks, emoji, and East Asian wide ranges without overfitting one test.
- ANSI erase semantics can be easy to regress because rows and cursor coordinates are tracked separately.
- Extending factory signatures can affect generated declaration output, so type checking must validate the public API.
- Posting replies before verification would misrepresent status, so replies should wait until tests pass.

## Assumptions

- All five P2 comments are valid and should be fixed in one simple review-fix tier.
- The acknowledgement comments are already resolved follow-ups and do not need replies.
- A local display-width implementation is preferable to adding a declared dependency for this limited runner behavior.
- No changeset is required because these are review fixes on an unreleased branch feature rather than a new user-facing change beyond the PR's existing scope.

## Open Questions

- None that change implementation scope; any unexpected test or typecheck failure should be resolved within the same review-fix scope.
