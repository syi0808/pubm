`pubm` est CLI-first, mais un vrai projet depasse vite les seuls flags. `pubm.config.*` permet de garder au niveau du depot les registres cibles, le layout des packages, les plugins et les regles de versionnement.

## Decouverte du fichier

1. `pubm.config.ts`
2. `pubm.config.mts`
3. `pubm.config.cts`
4. `pubm.config.js`
5. `pubm.config.mjs`
6. `pubm.config.cjs`

## Configuration minimale

`pubm` infere les registries depuis les fichiers de manifeste. La plupart des projets n'ont pas besoin d'un champ `registries`:

```ts
import { defineConfig } from "@pubm/core";

export default defineConfig({});
```

## Inference des registries

| Manifeste | Registry inferee |
|---|---|
| `package.json` | `npm` |
| `jsr.json` | `jsr` |
| `Cargo.toml` | `crates` |

## Config vs flags CLI

- la config fournit les valeurs par defaut pour `packages` et `plugins`
- `versioning`, `fixed`, `linked` sont utilises par les changesets
- les registries sont inferees; `packages[].registries` permet de les surcharger

## Exemple multi-package

Les registries sont inferees depuis les manifestes de chaque package:

```ts
export default defineConfig({
  versioning: "independent",
  packages: [
    { path: "packages/core" },
    { path: "packages/cli", registries: ["npm"] }, // surcharge
    { path: "crates/pubm-rs" },
  ],
});
```

## Strategies

- `independent`: chaque package evolue a son rythme
- `fixed`: un meme numero de version pour tous les packages concernes
- `fixed` / `linked`: groupes hybrides

Voir [Plugins API Reference](/pubm/fr/reference/plugins/) pour les plugins.
