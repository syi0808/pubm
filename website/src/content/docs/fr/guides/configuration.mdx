title: Configuration
description: Definir les valeurs par defaut du pipeline de release, la structure des packages et les plugins.
---

`pubm` est CLI-first, mais un vrai projet depasse vite les seuls flags. `pubm.config.*` permet de garder au niveau du depot les registres cibles, le layout des packages, les plugins et les regles de versionnement.

## Fichiers de configuration pris en charge

1. `pubm.config.ts`
2. `pubm.config.mts`
3. `pubm.config.cts`
4. `pubm.config.js`
5. `pubm.config.mjs`
6. `pubm.config.cjs`

## Valeurs par defaut

Le schema de configuration resolve notamment :

- `versioning: "independent"`
- `packages: [{ path: "." }]`
- `fixed: []`
- `linked: []`
- `tag: "latest"`
- `snapshotTemplate: "{tag}-{timestamp}"`

## Inference des registres

| Manifeste | Registre infere |
|---|---|
| `package.json` | `npm` |
| `jsr.json` | `jsr` |
| `Cargo.toml` | `crates` |

Regles npm supplementaires :

- `publishConfig.registry` remplace `npm`
- `.npmrc` remplace `npm` si `publishConfig` est absent

## Options importantes

- `packages` : declaration explicite des packages a publier
- `plugins` : hooks, commandes et extensions runtime
- `versioning`, `fixed`, `linked` : planification des versions
- `ignore` : exclusions de discovery
- `snapshotTemplate` : format des versions snapshot

## Configuration multi-package

```ts
export default defineConfig({
  versioning: "independent",
  packages: [
    { path: "packages/core" },
    { path: "packages/cli", registries: ["npm"] },
    { path: "crates/pubm-rs" },
  ],
});
```

Utilisez `registries` seulement quand l'inference n'est pas suffisante ou pour ajouter un registre prive.
