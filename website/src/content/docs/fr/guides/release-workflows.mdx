title: Workflows de release
description: Choisir entre Direct Release, Split CI Release et GitHub Release PRs pour pubm.

Utilisez ce guide pour choisir comment une vraie release passe des vérifications locales aux packages publiés.

`pubm` prend actuellement en charge trois workflows de release :

| Workflow | Commandes | À utiliser quand |
|----------|-----------|------------------|
| Direct Release | `pubm` | Un terminal local de confiance ou un job contrôlé doit exécuter toute la release. |
| Split CI Release | Local `pubm --phase prepare`, puis CI `pubm --phase publish` | La préparation locale doit écrire les versions et tags; CI doit publier avec des tokens. |
| GitHub Release PR | `pubm workflow install github` | Les changements de fichiers de release doivent être relus dans des PRs avant que CI publie les releases fusionnées. |

## Direct Release

Direct Release exécute tout le pipeline de release avec une seule commande :

```bash
pubm
```

Utilisez-le quand la personne ou le job qui démarre la release peut détenir les credentials de registry, pousser le commit de release et les tags, puis publier les packages en sécurité.

Direct Release :

- valide le projet et l'état Git
- exécute les vérifications préalables et les conditions requises
- exécute les tests et le build sauf s'ils sont ignorés
- consomme les changesets en attente lorsqu'ils existent
- écrit les versions des manifests
- crée le commit de release et les tags
- publie les packages vers les registries sélectionnés
- crée les GitHub Releases lorsque c'est activé
- annule les échecs côté publication lorsque c'est possible

## Split CI Release

Split CI Release sépare l'étape de préparation mutable de l'étape de publication non interactive :

```bash
pubm --phase prepare
```

```bash
pubm --phase publish
```

Utilisez-le lorsque les maintainers doivent approuver ou préparer la release localement, mais que la publication registry doit s'exécuter dans CI avec les secrets du dépôt.

### Prepare for CI publish

Exécutez Prepare for CI publish localement :

```bash
pubm --phase prepare
```

Prepare for CI publish :

- collecte les tokens de registry et les credentials de plugins
- synchronise éventuellement les credentials vers GitHub Secrets
- exécute les vérifications préalables et les conditions requises
- exécute les tests et le build sauf s'ils sont ignorés
- consomme les changesets en attente pendant le versioning lorsqu'ils existent
- écrit les versions des manifests
- crée le commit de release et les tags
- pousse le commit de release et les tags que CI doit reprendre
- exécute les tâches de publication en dry-run au lieu de publier réellement
- ne publie pas les packages

### Publish prepared release

Exécutez Publish prepared release dans CI :

```bash
pubm --phase publish
```

Publish prepared release :

- lit les versions déjà écrites dans les manifests de packages
- publie les packages vers les registries sélectionnés
- crée les GitHub Releases et téléverse les release assets lorsque c'est configuré
- attend les tokens de registry et de plugins depuis l'environnement CI
- s'exécute sans invites interactives
- ne refait pas le versioning
- ne recrée pas les tags
- ne repousse pas le commit de release ni les tags

## GitHub Release PR

GitHub Release PR utilise des sub-actions `pubm-actions` séparées :

```bash
pubm workflow install github
```

L'action `release-pr` écrit les versions de manifests, les changelogs, les changesets consommés et la sortie des hooks `afterVersion` dans une PR gérée. L'action `publish` s'exécute seulement après la fusion d'une release PR labellisée, crée les tags sur le merge commit, publie le scope correspondant et crée les GitHub Releases.

Les release PRs sont des pending PRs continues par release scope. Si d'autres changements publiables arrivent avant la fusion, pubm met à jour la PR existante de ce package ou groupe linked/fixed au lieu d'ouvrir un doublon pour la nouvelle version. Le branch template par défaut ne contient pas la version : `pubm/release/{scopeSlug}`.

## Portée actuelle

La CI entièrement automatisée, où CI décide et prépare la release sans `pubm --phase prepare` local, n'est pas un workflow actuellement pris en charge.
