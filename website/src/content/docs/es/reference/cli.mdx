Esta referencia se agrupa por forma de comando:

- el comando principal de publicaciﾃｳn: `pubm [version]`

## Comando principal de publicaciﾃｳn

### Sintaxis

```bash
pubm [version] [flags]
```

`[version]` puede ser:

- un tipo de bump semver: `patch`, `minor`, `major`
- un tipo de bump de pre-release: `prepatch`, `preminor`, `premajor`, `prerelease`
- una versiﾃｳn explﾃｭcita como `1.8.0`

Si omites la versiﾃｳn:

- las terminales interactivas pueden preguntar por la siguiente versiﾃｳn
- las Direct Release no interactivas necesitan una versiﾃｳn explﾃｭcita o una fuente de versiﾃｳn configurada
- Split CI Release publish lee las versiones ya preparadas por `pubm --phase prepare`

### Uso interactivo

```bash
pubm
```

Flujo tﾃｭpico:

1. ejecuta `pubm`
2. elige `patch`, `minor` o `major`
3. deja que `pubm` continﾃｺe con checks, build, tagging y publicaciﾃｳn

### Ejemplos comunes

```bash
pubm
pubm --dry-run
pubm 1.8.0
pubm --registry npm,jsr
pubm --tag beta
```

## Flags de publicaciﾃｳn

| Flag | Por defecto | Descripciﾃｳn |
|------|---------|-------------|
| `-d, --dry-run` | `false` | Muestra el grafo de tareas sin mutar el estado de Git ni publicar. |
| `--phase <phase>` | - | Ejecuta una fase de Split CI Release: `prepare` (Prepare for CI publish) o `publish` (Publish prepared release). Omﾃｭtelo para Direct Release. |
| `--release-draft` | `false` | Crea un GitHub Release en borrador en lugar de uno publicado. |
| `-b, --branch <name>` | `main` | Exige que el branch actual coincida antes de publicar. |
| `-a, --any-branch` | `false` | Desactiva la protecciﾃｳn por branch para la ejecuciﾃｳn actual. |
| `-t, --tag <name>` | `latest` | Usa un dist-tag especﾃｭfico como `beta` o `next`. |
| `-c, --contents <path>` | none | Cambia a un subdirectorio antes de ejecutar el pipeline de release. |
| `--registry <registries>` | `npm,jsr` | Registries destino separados por coma. Soporta `npm`, `jsr`, `crates` o URLs personalizadas de registry. |
| `--test-script <script>` | `test` | Nombre del script usado para el paso de test. |
| `--build-script <script>` | `build` | Nombre del script usado para el paso de build. |
| `--no-save-token` | `false` | No persiste los tokens compatibles para ejecuciones futuras. |
| `--dangerously-allow-unpublish` | `false` | Permite el unpublish/yank del registry durante el rollback en entornos sin TTY. |
| `--locale <locale>` | `en` | Establece el idioma de salida del CLI. Admitidos: `en`, `ko`, `zh-cn`, `fr`, `de`, `es`. |
| `--create-pr` | `false` | Crea un pull request para el version bump en lugar de hacer push directamente. |

## Flags de control del pipeline

| Flag | Descripciﾃｳn |
|------|-------------|
| `--no-pre-check` | Omite comprobaciones previas como la validaciﾃｳn de branch y working tree. |
| `--no-condition-check` | Omite comprobaciones de condiciones requeridas como la validaciﾃｳn de tokens y registries. |
| `--no-tests` | Omite el paso de tests. |
| `--no-build` | Omite el paso de build. |
| `--no-publish` | Ejecuta el pipeline hasta publish, pero no publica artefactos realmente. |
| `--no-dry-run-validation` | Omite la validaciﾃｳn de dry-run durante la fase de preparaciﾃｳn. ﾃ嗾il cuando crates de workspace tienen dependencias internas cuyas nuevas versiones aﾃｺn no se han publicado. |
| `--skip-release` | Omite la creaciﾃｳn de GitHub Release. |

## Workflows de release

`pubm` admite Direct Release y Split CI Release. Lee [Workflows de release](/pubm/es/guides/release-workflows/) para el comportamiento conceptual y cuﾃ｡ndo elegir cada ruta.

| Workflow | Forma del comando |
|----------|-------------------|
| Direct Release | `pubm` |
| Split CI Release prepare | `pubm --phase prepare` |
| Split CI Release publish | `pubm --phase publish` |

## Modo dry-run

```bash
pubm --dry-run
```

Usa el modo dry-run cuando necesites inspeccionar el plan de tareas sin efectos secundarios.

## Split CI Release: Prepare for CI publish

```bash
pubm --phase prepare
```

Prepare for CI publish:

- valida el proyecto, el acceso a registries y las credenciales de plugins
- recopila tokens faltantes y puede sincronizarlos con GitHub Secrets
- escribe versiones de manifests, crea release tags y hace push del commit de release y tags
- ejecuta comprobaciones previas y de condiciones
- simula la publicaciﾃｳn para los registries configurados
- no publica paquetes

## Split CI Release: Publish prepared release

```bash
pubm --phase publish
```

Publish prepared release lee versiones de manifests, publica paquetes y crea GitHub Releases mediante la API. Estﾃ｡ pensado para CI y ejecuciﾃｳn no interactiva con tokens despuﾃｩs de que Prepare for CI publish ya haya escrito versiones, creado tags y hecho push del commit de release y tags. Pasa `--release-draft` para crear releases en borrador en lugar de publicados.

## `pubm snapshot`

Publica versiones snapshot para previsualizaciﾃｳn y pruebas. Compatible con proyectos de paquete ﾃｺnico y monorepo.

```bash
pubm snapshot [tag]
```

| Opciﾃｳn | Descripciﾃｳn |
|--------|-------------|
| `[tag]` | Tag del snapshot (por defecto: `snapshot`) |
| `-f, --filter <name>` | Filtrar paquetes por nombre o ruta (repetible) |
| `-d, --dry-run` | Simular sin efectos secundarios |
| `--no-tests` | Omitir tests |
| `--no-build` | Omitir build |
| `-b, --branch <name>` | Rama destino (por defecto: `main`) |
| `-a, --any-branch` | Permitir publicar desde cualquier rama |

### Ejemplos

```bash
# Snapshot de todos los paquetes
pubm snapshot

# Snapshot con tag personalizado
pubm snapshot beta

# Snapshot de paquetes especﾃｭficos
pubm snapshot --filter @pubm/core --filter pubm
```

## `pubm init`

Asistente de configuraciﾃｳn interactivo para `pubm`. Requiere un TTY.

```bash
pubm init
```

El asistente pasa por estas etapas:

1. **Detecciﾃｳn de paquetes**: escanea el repositorio en busca de archivos manifest (`package.json`, `jsr.json`, `deno.json`, `deno.jsonc`, `Cargo.toml`) y la configuraciﾃｳn del workspace, y confirma los paquetes y registries detectados.
2. **Configuraciﾃｳn bﾃ｡sica**: solicita el branch de release y la estrategia de versionado (`independent` o `fixed`).
3. **Opciones de release**: pregunta sobre la generaciﾃｳn de changelog y los borradores de GitHub Release.
4. **Configuraciﾃｳn de workflows**: ofrece activar el workflow de changesets (crea `.pubm/changesets/`, `.github/workflows/pubm-changeset-check.yml` y actualiza `.gitignore`) y generar workflows de release para CI (`.github/workflows/pubm-release-pr.yml` and `.github/workflows/pubm-publish.yml`).
5. **Skills de agentes de cﾃｳdigo**: ofrece descargar e instalar skills de agentes de cﾃｳdigo para Claude Code, Codex CLI o Gemini CLI. Esto equivale a `pubm setup-skills`.

El archivo `pubm.config.ts` solo se crea cuando tus elecciones difieren de los valores por defecto. Si todos los valores coinciden con el comportamiento por defecto, no se escribe ningﾃｺn archivo de configuraciﾃｳn.

Este comando no acepta flags ni opciones: toda la configuraciﾃｳn se recopila de forma interactiva.

## `pubm workflow install github`

Install GitHub Actions workflows for the release PR workflow.

```bash
pubm workflow install github
```

| Option | Description |
| ------ | ----------- |
| `--force` | Overwrite existing generated workflow files |
| `--dry-run` | Print the files that would be written |
| `--package-manager <pm>` | Choose setup hints for `bun`, `pnpm`, `yarn`, `npm`, or `cargo` |
| `--branch <branch>` | Target branch for release PRs and publish guardrails |
| `--skip-changeset-check` | Do not install `.github/workflows/pubm-changeset-check.yml` |
| `--skip-release-pr` | Do not install `.github/workflows/pubm-release-pr.yml` |
| `--skip-publish` | Do not install `.github/workflows/pubm-publish.yml` |


| Flag | Por defecto | Descripciﾃｳn |

Notas:

- requiere que `gh` estﾃｩ instalado y autenticado
- usa tokens almacenados localmente, asﾃｭ que ejecuta primero `pubm --phase prepare` si todavﾃｭa no se ha guardado nada

## Comandos de changesets

Todos los comandos del ciclo de vida de changesets viven bajo `pubm changesets`.

## `pubm changesets add`

Crea un nuevo changeset.

```bash
pubm changesets add
pubm changesets add --empty
pubm changesets add --packages @acme/core,@acme/react --bump minor --message "Add public metrics API."
```

| Flag | Descripciﾃｳn |
|------|-------------|
| `--empty` | Crea un archivo de changeset vacﾃｭo. |
| `--packages <list>` | Nombres de paquetes separados por comas para creaciﾃｳn no interactiva. |
| `--bump <type>` | Uno de `patch`, `minor`, `major`. |
| `--message <text>` | Resumen usado en el cuerpo del changeset y en el changelog. |

## `pubm changesets status`

Inspecciona los changesets pendientes.

```bash
pubm changesets status
pubm changesets status --verbose
pubm changesets status --since origin/main
```

| Flag | Descripciﾃｳn |
|------|-------------|
| `--verbose` | Imprime el resumen de cada changeset registrado. |
| `--since <ref>` | Comprueba desde un ref de Git. |

## `pubm changesets version`

Consume changesets, actualiza las versiones de los manifests, escribe changelogs y hace commit del resultado.

`pubm changesets version` analyzes pending changesets and conventional commits together. Changesets win when both sources cover the same package. See the [Configuration guide](/pubm/es/guides/configuration/#release-analysis) for `release.changesets` and `release.commits` options.

```bash
pubm changesets version
pubm changesets version --dry-run
```

| Flag | Descripciﾃｳn |
|------|-------------|
| `--dry-run` | Imprime el plan de versiones y changelog sin escribir archivos. |

## `pubm changesets changelog`

Genera texto de changelog a partir de changesets pendientes.

```bash
pubm changesets changelog
pubm changesets changelog --dry-run
pubm changesets changelog --version 1.8.0
```

| Flag | Descripciﾃｳn |
|------|-------------|
| `--dry-run` | Previsualiza la salida del changelog sin escribir `CHANGELOG.md`. |
| `--version <ver>` | Usa un encabezado de secciﾃｳn concreto en lugar de `Unreleased`. |

## `pubm changesets migrate`

Mueve archivos pendientes del layout heredado `.changeset/` a `.pubm/`.

```bash
pubm changesets migrate
```

## Cﾃｳdigos de salida

`pubm` devuelve un cﾃｳdigo de salida distinto de cero cuando:

- falla una comprobaciﾃｳn previa o de condiciones
- no se puede completar la resoluciﾃｳn de versiﾃｳn
- falla la publicaciﾃｳn en cualquier target
- un subcomando encuentra entrada no vﾃ｡lida

En el modo de publicaciﾃｳn normal, los fallos tambiﾃｩn desencadenan rollback cuando es posible.
