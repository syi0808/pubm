title: Changesets

`pubm` usa changesets como base para el versionado y el changelog. Cada changeset describe qué paquetes cambian y en qué nivel: patch, minor o major.

## Flujo básico

```bash
pubm changesets add
pubm changesets status
pubm changesets version
pubm changesets changelog
```

## Inicializar el directorio de changesets

Los changesets pendientes se almacenan en:

```text
.pubm/changesets/
```

Inicializa ese directorio ejecutando el asistente de configuración interactivo:

```bash
pubm init
```

Durante el setup, `pubm init` pregunta si deseas activar el workflow de changesets. Si seleccionas que sí, crea el directorio `.pubm/changesets/`, genera `.github/workflows/pubm-changeset-check.yml` para aplicar el requisito en PRs, y actualiza `.gitignore` para registrar los archivos de changeset ignorando el resto del contenido de `.pubm/`.

## Añadir un changeset

```bash
pubm changesets add
```

Ese comando guía el proceso y crea un archivo en `.pubm/changesets/`.

## Ver estado

```bash
pubm changesets status
pubm changesets status --verbose
```

`status` muestra qué changesets siguen pendientes y qué bumps producirán.

## Consumir changesets

```bash
pubm changesets version
```

`version` aplica los changesets pendientes a los paquetes afectados y escribe las nuevas versiones.

## Generar changelog

```bash
pubm changesets changelog
```

`changelog` genera texto de changelog a partir de los changesets pendientes.

## Uso en CI

Si tu release se ejecuta en CI, no necesitas un paso separado de `pubm changesets version` para preparar la publicación. El pipeline normal de `pubm` ya consume los changesets durante la fase de versionado.

```bash
pubm
pubm --phase publish
```

Ese comando es útil para flujos editoriales o controlados, pero no es necesario antes de `pubm` ni de `pubm --phase publish`.

## Conventional commit recommendations

`pubm changesets version` analyzes conventional commits alongside pending changeset files. If a package has no changeset recommendation, a releasable conventional commit can create the bump recommendation. Non-conventional commits, ignored commit types, and commits that cannot be mapped to a package are treated as unversioned changes; they do not create bumps by themselves.

### Default commit type mapping

| Commit type | Bump |
| --- | --- |
| `feat` | `minor` |
| `fix` | `patch` |
| `perf` | `patch` |

A `BREAKING CHANGE` footer or a `!` suffix (e.g. `feat!:`) always produces a `major` bump.

### Configuring custom mappings

Override the default mapping in `pubm.config.ts`:

```ts
export default defineConfig({
  release: {
    commits: {
      types: {
        feat: "minor",
        fix: "patch",
        perf: "patch",
        docs: "patch",
      },
    },
  },
});
```

Release PR automation warns by default when only unversioned changes are found and no versioned release PR can be opened. Configure that with `release.pullRequest.unversionedChanges`: `"warn"` (default), `"ignore"`, or `"fail"`.

## Migrar desde changesets

```bash
pubm changesets migrate
```

`migrate` ayuda a mover archivos desde `.changeset/` a `.pubm/`.
