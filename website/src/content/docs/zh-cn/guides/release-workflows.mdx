title: 发布工作流
description: 在 pubm 中选择 Direct Release、Split CI Release 或 GitHub Release PR。

使用本指南来决定一次真实 release 如何从本地检查推进到已发布的 packages。

`pubm` 当前支持三种发布工作流：

| 工作流 | 命令 | 适用场景 |
|--------|------|----------|
| Direct Release | `pubm` | 一个受信任的本地终端或受控 job 应运行完整 release |
| Split CI Release | 本地 `pubm --phase prepare`，然后 CI `pubm --phase publish` | 本地准备应写入版本和 tags，CI 应使用 tokens 完成发布 |
| GitHub Release PR | `pubm workflow install github` | release 文件变更需要先在 PR 中审查，然后 CI 发布已合并的 release |

## Direct Release

Direct Release 用一个命令运行完整 release pipeline：

```bash
pubm
```

当启动 release 的人员或 job 可以安全持有 registry credentials、推送 release commit 和 tags 并发布 packages 时使用。

Direct Release 会：

- 验证项目和 Git 状态
- 运行 prerequisite 和 required-condition 检查
- 运行测试和构建，除非被跳过
- 在存在待处理 changeset 时消耗它们
- 写入 manifest versions
- 创建 release commit 和 tags
- 发布 packages 到选定 registries
- 在启用时创建 GitHub Releases
- 在可能时 rollback publish-side failures

## Split CI Release

Split CI Release 将可变的准备步骤和非交互式发布步骤分开：

```bash
pubm --phase prepare
```

```bash
pubm --phase publish
```

当 maintainers 应在本地批准或准备 release，但 registry publishing 应在拥有 repository secrets 的 CI 中执行时使用。

### Prepare for CI publish

在本地运行 Prepare for CI publish：

```bash
pubm --phase prepare
```

Prepare for CI publish 会：

- 收集 registry tokens 和 plugin credentials
- 可选地将 credentials 同步到 GitHub Secrets
- 运行 prerequisite 和 required-condition 检查
- 运行测试和构建，除非被跳过
- 在存在待处理 changeset 时于 versioning 阶段消耗它们
- 写入 manifest versions
- 创建 release commit 和 tags
- 推送 CI 可读取的 release commit 和 tags
- dry-run publish tasks，而不执行真实发布
- 不发布 packages

### Publish prepared release

在 CI 中运行 Publish prepared release：

```bash
pubm --phase publish
```

Publish prepared release 会：

- 读取已经写入 package manifests 的 versions
- 发布 packages 到选定 registries
- 在配置时创建 GitHub Releases 并上传 release assets
- 期望 registry 和 plugin tokens 来自 CI 环境
- 不使用 interactive prompts
- 不重新执行 versioning
- 不再次创建 tags
- 不再次推送 release commit 或 tags

## GitHub Release PR

GitHub Release PR 使用拆分的 `pubm-actions` sub-actions：

```bash
pubm workflow install github
```

`release-pr` action 会把 manifest versions、changelogs、已消费的 changesets 和 `afterVersion` hook 输出写入受管理的 PR。`publish` action 只会在带 label 的 release PR 合并后运行，在 merge commit 上创建 tags，发布匹配的 scope，并创建 GitHub Releases。

Release PR 是按 release scope 持续更新的 pending PR。如果 PR 合并前又有可发布变更进入 release branch，pubm 会更新同一个 package 或 linked/fixed group 的现有 PR，而不是为新版本打开重复 PR。默认 release PR branch template 不包含版本：`pubm/release/{scopeSlug}`。

## 当前范围

完全自动化 CI，也就是 CI 在没有本地 `pubm --phase prepare` 的情况下决定并准备 release，目前不是受支持的工作流。
