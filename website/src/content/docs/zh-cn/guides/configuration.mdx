`pubm` 有意保持 CLI-first，但真实项目很快就会超出只靠 flags 配置的范围。`pubm.config.*` 用来保存仓库级的发布目标、包布局、插件和与 changeset 相关的版本规则。

运行 `pubm init` 可以通过交互方式生成此配置文件。只有当你的选择与下方所示默认值不同时，向导才会创建配置文件。

## 支持的配置文件

`pubm` 支持这些文件名：

1. `pubm.config.ts`
2. `pubm.config.mts`
3. `pubm.config.cts`
4. `pubm.config.js`
5. `pubm.config.mjs`
6. `pubm.config.cjs`

TypeScript 配置文件可直接使用。loader 会在执行前先打包该文件，因此不需要单独的运行时转译步骤。

## 默认配置

These are the default user-facing settings.

```ts
export default defineConfig({
  branch: "main",
  commit: false,
  access: "public",
  release: {
    versioning: {
      mode: "independent",
      fixed: [],
      linked: [],
      updateInternalDependencies: "patch",
    },
    changesets: {
      directory: ".pubm/changesets",
    },
    commits: {
      format: "conventional",
      types: {},
    },
    changelog: true,
    pullRequest: {
      branchTemplate: "pubm/release/{scopeSlug}",
      titleTemplate: "chore(release): {scope} {version}",
      label: "pubm:release-pr",
      bumpLabels: {
        patch: "release:patch",
        minor: "release:minor",
        major: "release:major",
        prerelease: "release:prerelease",
      },
      unversionedChanges: "warn",
    },
  },
  ignore: [],
  tag: "latest",
  contents: ".",
  saveToken: true,
  releaseDraft: true,
  releaseNotes: true,
  rollback: {
    strategy: "individual",
    dangerouslyAllowUnpublish: false,
  },
  packages: [{ path: "." }],
  validate: {
    cleanInstall: true,
    entryPoints: true,
    extraneousFiles: true,
  },
  snapshotTemplate: "{tag}-{timestamp}",
  locale: "en",
});
```

## registry 是如何推断的

`pubm` 会检查每个 package 目录中的 manifest 文件，并将它们映射到对应的 registry：

| Manifest | 推断出的 registry |
|---|---|
| `package.json` | `npm` |
| `jsr.json` | `jsr` |
| `deno.json` / `deno.jsonc` | `jsr` |
| `Cargo.toml` | `crates` |

npm 还适用两条额外规则：

- `package.json` 中的 `publishConfig.registry` 会把 `npm` 替换成自定义 URL
- 当没有 `publishConfig` 时，`.npmrc` 中的 registry 会替换 `npm`

## 顶层选项

### `release`

- Type: `ReleaseConfig`

Configures version planning, changeset discovery, conventional commit parsing, changelog generation, and Release PR automation.

#### `release.versioning`

Controls how versions are assigned in multi-package repositories.

| Key | Type | Default | Description |
| --- | --- | --- | --- |
| `mode` | `"independent" \| "fixed"` | `"independent"` | Version packages independently or as one release. |
| `fixed` | `string[][]` | `[]` | Package groups that must resolve to the same version. |
| `linked` | `string[][]` | `[]` | Package groups that share the highest bump type. |
| `updateInternalDependencies` | `"patch" \| "minor"` | `"patch"` | Minimum bump for internal dependency ranges. |

#### `release.changesets`

| Key | Type | Default | Description |
| --- | --- | --- | --- |
| `directory` | `string` | `.pubm/changesets` | Directory for pending changeset files. |

#### `release.commits`

Configures conventional commit parsing. Commit analysis is always available; use `types` to decide which commit types create semver bump recommendations.

| Key | Type | Default | Description |
| --- | --- | --- | --- |
| `format` | `string` | `"conventional"` | Commit parser format. |
| `types` | `Record<string, "patch" \| "minor" \| "major" \| false>` | `{}` | Overrides for commit type to semver bump mapping. |

#### `release.changelog`

- Type: `boolean`
- Default: `true`

Whether pubm should write changelog entries when materializing release versions.

#### `release.pullRequest`

Configuration for the GitHub release PR workflow installed by `pubm workflow install github`.

- `branchTemplate` defaults to `pubm/release/{scopeSlug}` so one pending release PR is maintained per package or group scope while its computed version changes.
- `titleTemplate` defaults to `chore(release): {scope} {version}`.
- Release PR dry-run validation always runs before merge.
- `grouping` inherits `release.versioning.mode` by default and can be overridden with `"fixed"` or `"independent"`.
- `fixed` and `linked` inherit `release.versioning.fixed` and `release.versioning.linked` by default.
- Leave `grouping`, `fixed`, and `linked` unset to inherit. Setting `fixed: []` or `linked: []` under `release.pullRequest` intentionally disables inherited groups for release PRs.
- `unversionedChanges` defaults to `"warn"`. Use `"ignore"` to suppress the warning or `"fail"` to fail Release PR automation when only unversioned changes are found.

### `branch`

- 类型：`string`
- 默认值：`"main"`

受保护的发布运行中，`HEAD` 必须匹配的分支。

### `packages`

- 类型：`PackageConfig[]`
- 默认值：`[{ path: "." }]`

显式声明可发布的包。除非在单个包上覆盖，否则 registry 会从 manifest 文件中推断。

### `plugins`

- 类型：`PubmPlugin[]`
- 默认值：`[]`

注册插件，这些插件可以添加 hooks、commands、registries 或 ecosystems。

### `ignore`

- 类型：`string[]`
- 默认值：`[]`

从 workspace 发现中排除的包名或路径。

### `tag`

- 类型：`string`
- 默认值：`"latest"`

发布运行使用的默认 dist-tag。

### `contents`

- 类型：`string`
- 默认值：`"."`

在运行发布流程之前切换到的目录。

### `saveToken`

- 类型：`boolean`
- 默认值：`true`

是否将受支持 registry 的 token 保存起来，供本地机器后续复用。

### `skipDryRun`

- Type: `boolean`
- Default: `false`

在准备阶段跳过 dry-run 验证。当工作区 crates 具有尚未在 crates.io 上发布新版本的内部依赖项时非常有用。也可以通过 `--no-dry-run-validation` CLI 标志设置。

### `releaseDraft`

- 类型：`boolean`
- 默认值：`true`

发布流程是否应该创建 GitHub Release。禁用后会完全跳过 release 步骤。

### `excludeRelease`

- 类型：`string[]`
- 默认值：`[]`

用于指定包的 glob 模式，这些包会被版本化和发布，但会被排除在 git tag 创建和 GitHub Release 草稿之外。仅在 `release.versioning.mode` 为 `"independent"` 时生效。

适用于内部构建产物，例如平台特定的二进制文件，它们需要发布到注册表，但不应以独立 release 的形式出现。

```ts
export default defineConfig({
  release: {
    versioning: {
      mode: "independent",
    },
  },
  excludeRelease: ["packages/cli/platforms/*"],
  packages: [
    { path: "packages/core" },
    { path: "packages/cli" },
    { path: "packages/cli/platforms/*" },
  ],
});
```

### `validate`

- 类型：`ValidateConfig`

这些字段属于公开 schema，但目前还不是所有 publish 路径上的活动开关。应把它们视为带类型的配置面，而不是保证在运行时生效的切换项。

### `snapshotTemplate`

- 类型：`string`
- 默认值：`"{tag}-{timestamp}"`

当 `pubm --snapshot` 生成临时版本字符串时使用的模板。

`package.json` 中的基础版本始终会自动添加到快照版本的前面。模板只控制基础版本之后的后缀部分。

可用的模板变量：

- `{tag}`：snapshot tag，默认是 `snapshot`
- `{timestamp}`：格式为 `YYYYMMDDTHHmmss` 的 UTC 时间戳
- `{commit}`：当前 git commit SHA

### `registryQualifiedTags`

- **Type:** `boolean`
- **Default:** `false`

When set to `true`, git tags in independent versioning mode are prefixed with the registry name to avoid collisions when packages from different ecosystems share the same name.

```ts
export default defineConfig({
  registryQualifiedTags: true,
  packages: [
    { path: 'js', registries: ['npm'] },
    { path: 'crates/cli', registries: ['crates'] },
  ],
})
```

Without this option, both packages named `my-tool` would produce the tag `my-tool@1.0.0`. With it enabled, tags become `npm/my-tool@1.0.0` and `crates/my-tool@1.0.0`.

pubm automatically detects name collisions and prompts you to enable this option. In CI environments, the collision causes an error with instructions to set this config option.

### `locale`

- 类型：`"en" | "ko" | "zh-cn" | "fr" | "de" | "es"`
- 默认值：`"en"`

CLI 输出语言。也可通过 `--locale` flag 或 `PUBM_LOCALE` 环境变量设置。解析顺序：`--locale` flag → `PUBM_LOCALE` 环境变量 → config → 系统语言 → `"en"`。

### `rollback`

控制发布失败时的回滚行为。

| 键 | 类型 | 默认值 | 描述 |
|----|------|--------|------|
| `strategy` | `"individual"` \| `"all"` | `"individual"` | 回滚粒度。取代已废弃的顶层字段 `rollbackStrategy`。 |
| `dangerouslyAllowUnpublish` | `boolean` | `false` | 允许在非 TTY（CI）环境中的回滚期间对 registry 执行 unpublish/yank。即使 unpublish 后，npm 也会永久保留该版本号，无法被重用。 |

```ts
export default defineConfig({
  rollback: {
    strategy: "individual",
    dangerouslyAllowUnpublish: true, // 在 CI 中启用
  },
});
```

已废弃的顶层字段 `rollbackStrategy` 仍然有效，并映射到 `rollback.strategy`。如果同时设置了两者，`rollback.strategy` 优先生效。

## 配置和 CLI flags 如何交互

当前的运行时行为是：

- config 提供 `packages` 和 `plugins` 的仓库默认值
- release planning config such as `release.versioning`, `release.changesets`, and `release.commits` is used by version and release PR flows
- CLI flags 仍然控制每次运行的发布行为
- registry 会从 manifest 推断；`packages[].registries` 可以覆盖该推断

如果要针对某个 registry 做一次性发布：

```bash
pubm --registry npm --tag beta
```

## 多包配置

对于 workspace 或混合语言仓库，请显式声明 package。registry 会从每个 package 的 manifest 文件中推断：

```ts
import { defineConfig } from "@pubm/core";

export default defineConfig({
  release: {
    versioning: {
      mode: "independent",
    },
  },
  packages: [
    {
      path: "packages/core",
    },
    {
      path: "packages/cli",
      registries: ["npm"],
    },
    {
      path: "crates/pubm-rs",
    },
  ],
});
```

在以下情况下建议使用显式 package 条目：

- 你只想让 `pubm` 发布发现到的 workspace 包中的一部分
- JavaScript 和 Rust 包共存于同一个仓库
- 你需要覆盖推断出的 registry，或者添加私有 registry

### `PackageConfig`

package 级条目长这样：

```ts
interface PackageConfig {
  path: string;
  registries?: Array<"npm" | "jsr" | "crates" | PrivateRegistryConfig>;
  ecosystem?: "js" | "rust";
  buildCommand?: string;
  testCommand?: string;
}

interface PrivateRegistryConfig {
  url: string;
  token: { envVar: string };
}
```

只有在推断不够用，或者你需要在内建 registry 之外再加私有 registry 时，才使用 `registries`：

```ts
packages: [
  {
    path: "packages/a",
    registries: [
      "npm",
      { url: "https://npm.internal.com", token: { envVar: "INTERNAL_NPM_TOKEN" } },
    ],
  },
];
```

## 版本策略

### 独立版本

每个包都有自己的版本和 tag。

```ts
export default defineConfig({
  release: {
    versioning: {
      mode: "independent",
    },
  },
});
```

### 固定版本

所有受影响的包共享一个版本。

```ts
export default defineConfig({
  release: {
    versioning: {
      mode: "fixed",
    },
  },
});
```

### fixed 和 linked 分组

你可以让大多数包保持独立，同时强制选定的包一起移动：

```ts
export default defineConfig({
  release: {
    versioning: {
      mode: "independent",
      fixed: [["@acme/core", "@acme/react"]],
      linked: [["@acme/cli", "@acme/config"]],
    },
  },
});
```

- `fixed`：分组内所有包最终会使用完全相同的版本
- `linked`：最高的 bump 类型会在整个分组中传播

## Snapshot 设置

Snapshot 适合预览发布和按分支验证。

```ts
export default defineConfig({
  snapshotTemplate: "{tag}-{timestamp}",
});
```

基础版本会自动添加到前面，因此 `"{tag}-{timestamp}"` 模板会产生如下输出：

- `0.4.1-snapshot-20260312T102233`

## 插件

通过 `plugins` 数组添加插件：

```ts
import { defineConfig } from "@pubm/core";
import { externalVersionSync } from "@pubm/plugin-external-version-sync";

export default defineConfig({
  plugins: [
    externalVersionSync({
      targets: [
        {
          file: "README.md",
          pattern: /pubm@([\\w.-]+)/,
        },
      ],
    }),
  ],
});
```

阅读 [Plugins API Reference](/pubm/zh-cn/reference/plugins/) 了解 hooks 顺序、命令注册和自定义 registry 支持。

## 示例模式

### 简单库

如果包里有 `package.json` 和 `jsr.json`（或 `deno.json` / `deno.jsonc`），registry 会自动推断。不需要配置文件，`pubm.config.ts` 是可选的。只有在你需要覆盖默认值时才创建它，例如自定义 branch、plugins、release assets 等。

### App monorepo

只有 `package.json` 的包会被解析为 `npm`。除非你想覆盖，否则不需要显式 `registries`。

```ts
import { defineConfig } from "@pubm/core";

export default defineConfig({
  release: {
    versioning: {
      mode: "fixed",
    },
  },
  packages: [
    { path: "packages/web" },
    { path: "packages/server" },
  ],
});
```

### 混合 JS + Rust workspace

```ts
import { defineConfig } from "@pubm/core";

export default defineConfig({
  release: {
    versioning: {
      mode: "independent",
    },
  },
  packages: [
    { path: "packages/sdk" },
    { path: "crates/sdk-core" },
  ],
});
```

## Release 资产

`releaseAssets` 用来声明每个 GitHub Release 要附带的二进制构件。大多数情况下，一个简单的 glob 字符串就足够了：

```ts
import { defineConfig } from "@pubm/core";

export default defineConfig({
  releaseAssets: [
    "platforms/*/bin/mytool",
  ],
});
```

### `compress`

- 类型：`"tar.gz" | "zip" | "tar.xz" | "tar.zst" | false | Record<string, CompressFormat>`
- 默认值：按 OS 自动检测（windows → `zip`，其他 → `tar.gz`）

全局默认压缩格式。可以在 `releaseAssets` 内的 group 或 file 级别覆盖。

```ts
export default defineConfig({
  // Windows 使用 zip，其他平台使用 tar.gz
  compress: { windows: "zip" },

  releaseAssets: [
    "platforms/*/bin/mytool",
  ],
});
```

### `releaseAssets`

- 类型：`(string | ReleaseAssetGroupConfig)[]`
- 默认值：`undefined`（不附加任何资产）

每一项要么是 glob 字符串，要么是带显式控制的对象：

```ts
releaseAssets: [
  // 字符串：glob，全自动
  "platforms/*/bin/mytool",

  // 对象：按 group 显式配置
  {
    packagePath: "packages/cli",    // monorepo：把资产关联到这个 package 的 Release
    files: [
      "platforms/*/bin/mytool",
      { path: "dist/*.dmg", compress: false, name: "MyTool-{version}-{os}" },
    ],
    compress: "tar.gz",             // group 默认值
    name: "{name}-{platform}",      // group 名称模板
  },
]
```

阅读 [Release Assets](/pubm/zh-cn/guides/release-assets/) 了解完整的 compress cascade 规则、name 模板变量和使用场景示例。

## Release analysis

`pubm changesets version` analyzes pending changeset files and conventional commit messages together. Changeset recommendations win when a changeset and a commit both cover the same package. Non-conventional commits, ignored commit types, and unmatched package changes are treated as unversioned changes; they never create version bumps by themselves.

Use `release.changesets.directory` to move the changeset file directory. Use `release.commits.types` to control which conventional commit types create semver bump recommendations.

```ts
export default defineConfig({
  release: {
    changesets: {
      directory: ".pubm/changesets",
    },
    commits: {
      types: {
        // Add types that are ignored by default
        refactor: "patch",
        docs: "patch",
      },
    },
  },
});
```

The default mapping is:

| Commit type | Bump |
| --- | --- |
| `feat` | `minor` |
| `fix` | `patch` |
| `perf` | `patch` |

Commits with `BREAKING CHANGE` in the footer or a `!` suffix (e.g. `feat!:`) always produce a `major` bump regardless of the type mapping.

Commit types not listed in the mapping (e.g. `chore`, `docs`, `test`, `ci`, `style`, `refactor`) are ignored during bump calculation. You can override any type in `release.commits.types`; for example, `refactor: "patch"` would treat refactor commits as patch bumps.

When Release PR automation finds only unversioned changes, `release.pullRequest.unversionedChanges` controls the outcome. The default is `"warn"`, which reports the unversioned changes and opens no PR. `"ignore"` suppresses the warning. `"fail"` fails the automation.
