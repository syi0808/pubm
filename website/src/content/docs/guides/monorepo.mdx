title: Monorepo

`pubm` treats monorepos as a core use case, not a plugin scenario. It discovers packages, understands dependency order, and can release JavaScript and Rust packages from the same repository.

## What pubm detects automatically

Workspace discovery currently looks for:

- `pnpm-workspace.yaml`
- `package.json` with a `workspaces` field

After discovery, `pubm` builds a dependency graph and uses it to determine publish order.

## When to rely on discovery vs. explicit package config

Automatic discovery works well when:

- every workspace package is publishable
- every package targets the same registries
- you do not need to mix ecosystems

Use explicit `packages` config when:

- only some workspace packages should publish
- registry targets differ by package
- Rust crates live alongside JS packages
- you want the docs and release pipeline to stay explicit

```ts
import { defineConfig } from "@pubm/core";

export default defineConfig({
  packages: [
    { path: "packages/core", registries: ["npm", "jsr"], ecosystem: "js" },
    { path: "packages/cli", registries: ["npm"], ecosystem: "js" },
    { path: "crates/pubm-rs", registries: ["crates"], ecosystem: "rust" },
  ],
});
```

## Publish order

For dependent packages, `pubm` publishes in topological order.

If:

- `@acme/core` is a dependency of `@acme/react`
- `crate-a` is a dependency of `crate-b`

then `pubm` publishes the dependency before the dependent package.

This matters because:

- npm and jsr consumers can install the new dependency version immediately
- crates.io publishing must respect crate dependency ordering
- partially ordered publishes are easier to roll back and diagnose

## Circular dependencies

If a circular dependency prevents a safe publish order, `pubm` fails instead of guessing. Fix the workspace dependency graph first; release tooling should not be responsible for masking architectural cycles.

## Versioning modes

## `independent`

`independent` is the default mode.

```ts
export default defineConfig({
  versioning: "independent",
});
```

Behavior:

- each package can receive a different version
- each package is tagged independently
- changesets affect only the packages they mention unless groups change that result

This mode fits large library suites and plugin ecosystems.

## `fixed`

```ts
export default defineConfig({
  versioning: "fixed",
});
```

Behavior:

- packages released together share one version
- the repository gets one release tag such as `v1.8.0`
- changelog generation is simpler for product-style monorepos

This mode fits app platforms and tightly coupled packages.

## Fixed and linked groups

Hybrid monorepos usually need more nuance than a single global mode.

### Fixed groups

```ts
export default defineConfig({
  versioning: "independent",
  fixed: [["@acme/core", "@acme/react", "@acme/vue"]],
});
```

A fixed group keeps specific packages on exactly the same version even when the rest of the repository remains independent.

### Linked groups

```ts
export default defineConfig({
  versioning: "independent",
  linked: [["@acme/cli", "@acme/config"]],
});
```

A linked group propagates the highest bump type through the group. If one package receives a `minor`, the others also receive at least a `minor`.

## Typical release flow in a monorepo

```bash
pubm init --changesets
pubm changesets add
pubm changesets status --verbose
pubm changesets version --dry-run
pubm changesets version
pubm patch --preview
pubm patch
```

What changes between single-package and monorepo releases is not the command shape, but the output:

- `changesets status` shows per-package bump summaries
- `changesets version` writes multiple manifest versions when needed
- `pubm patch` tags either the repository version or each package version depending on the selected strategy

## Changesets in a monorepo

The most important rule is simple: put packages in the changeset file, not in commit messages or PR titles.

`pubm changesets add` will:

- discover available packages
- let you choose one or more packages
- prompt for the bump type for each package
- record the release summary

That information then drives:

- version bump calculation
- changelog generation
- fixed and linked group propagation

## Multi-registry monorepos

It is common for packages in one workspace to publish to different targets:

```ts
export default defineConfig({
  packages: [
    { path: "packages/sdk", registries: ["npm", "jsr"], ecosystem: "js" },
    { path: "packages/cli", registries: ["npm"], ecosystem: "js" },
    { path: "crates/runtime", registries: ["crates"], ecosystem: "rust" },
  ],
});
```

During publish, `pubm` groups work by ecosystem and registry:

- JavaScript packages can publish to npm and jsr concurrently
- Rust crates are published sequentially in dependency order
- plugin-provided registries are added to the same pipeline

## Git tags in monorepos

Tag format depends on the resulting version plan.

### Fixed mode

```text
v1.4.0
```

### Independent mode

```text
@acme/core@1.4.0
@acme/react@1.2.0
```

This distinction matters when you wire GitHub releases, changelog parsing, or downstream automation to tags.

## Publishing only selected registries

Even in a monorepo, registry selection can be overridden per run:

```bash
pubm patch --registry npm
pubm patch --registry npm,jsr
```

Use this carefully. A repository-level version bump still happened before publish, so selective publishing is best reserved for controlled workflows such as staged rollouts or emergency recovery.

## Operational guidance

Recommended defaults for most teams:

- use `pubm init --changesets` on day one
- keep packages explicit in config if registry targets differ
- prefer `independent` for libraries and `fixed` for products
- run `pubm --preflight` before enabling CI publish
- keep changelog ownership close to the package by writing good changeset summaries

## Common failure cases

### A package is discovered but should not publish

Move to explicit `packages` config or add the package name/path to `ignore`.

### Tags look different after switching versioning strategy

That is expected. `fixed` and `independent` produce different tagging schemes.

### Rust crates publish in a different order than JS packages

That is also expected. Crates are published sequentially to respect crate dependency order.
