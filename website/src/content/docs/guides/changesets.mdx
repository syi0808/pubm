---
title: Changesets
description: Record release intent, generate changelogs, and control version bumps.
---

`pubm` uses changesets as the checked-in record of what should be released next. It reads files from `.pubm/changesets/` and turns them into version bumps and changelog entries. It also analyzes conventional commits, but changesets win when both sources cover the same package.

## Why changesets matter

- a reviewable release plan in each pull request
- deterministic version bump calculation across packages
- changelog content written before release day, not after it

## File location

Pending changesets live in:

```text
.pubm/changesets/
```

Initialize that directory by running the interactive setup wizard:

```bash
pubm init
```

During setup, `pubm init` asks whether to enable the changesets workflow. If you choose yes, it creates the `.pubm/changesets/` directory, generates `.github/workflows/pubm-changeset-check.yml` for PR enforcement, and updates `.gitignore` so changeset files are tracked while other `.pubm/` contents stay ignored.

## Creating a changeset

```bash
pubm changesets add
```

In a monorepo, `pubm` also asks which packages are affected and what bump type each one should receive.

For non-interactive use:

```bash
pubm changesets add \
  --packages @acme/core,@acme/react \
  --bump minor \
  --message "Add token-based auth to the public client."
```

## What a changeset file contains

```md
---
"@acme/core": minor
"@acme/react": patch
---

Add token-based auth to the public client.
```

The metadata controls version calculation. The body becomes the changelog summary.

## Inspecting pending changes

Inspect changesets before `pubm` consumes them:

```bash
pubm changesets status
pubm changesets status --verbose
pubm changesets version --dry-run
```

## How changesets are consumed

When you run the normal release pipeline with `pubm`, pending changesets are consumed during the version step.

The flow is:

1. reads pending `.pubm/changesets/*.md` files
2. calculates the resulting package versions
3. updates manifest versions
4. generates changelog entries
5. deletes consumed changeset files
6. creates the release commit and tags

If you want to run only the versioning step ahead of time, you can still use:

```bash
pubm changesets version
```

That command is useful for editorial or controlled workflows, but it is not required before `pubm` or `pubm --phase prepare`. In Split CI Release, `pubm --phase publish` assumes the prepare phase has already consumed changesets and written manifest versions.

## Changelog generation

If you want to generate changelog text without consuming versions yet:

```bash
pubm changesets changelog --dry-run
pubm changesets changelog --version 1.8.0
```

## Conventional commit recommendations

`pubm changesets version` analyzes conventional commits alongside pending changeset files. If a package has no changeset recommendation, a releasable conventional commit can create the bump recommendation. Non-conventional commits, ignored commit types, and commits that cannot be mapped to a package are treated as unversioned changes; they do not create bumps by themselves.

### Default commit type mapping

| Commit type | Bump    |
| ----------- | ------- |
| `feat`      | `minor` |
| `fix`       | `patch` |
| `perf`      | `patch` |

A `BREAKING CHANGE` footer or a `!` suffix (e.g. `feat!:`) always produces a `major` bump.

### Configuring custom mappings

Override the default mapping in `pubm.config.ts`:

```ts
export default defineConfig({
  release: {
    commits: {
      types: {
        feat: "minor",
        fix: "patch",
        perf: "patch",
        docs: "patch",
      },
    },
  },
});
```

Release PR automation warns by default when only unversioned changes are found and no versioned release PR can be opened. Configure that with `release.pullRequest.unversionedChanges`: `"warn"` (default), `"ignore"`, or `"fail"`.

## Migration from `.changeset/`

```bash
pubm changesets migrate
```

This migrates pending files from `.changeset/` into `.pubm/`.
