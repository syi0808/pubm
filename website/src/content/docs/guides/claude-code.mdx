`pubm` is a good fit for agent-assisted release work because the core workflow is explicit: inspect state, preview the plan, then publish. That matches how Claude Code and similar tools are typically used in real repositories.

## What this guide covers

This page describes the intended integration model:

- expose release workflows as explicit commands or skills
- keep preview and validation steps first-class
- let the agent gather release context without guessing

## Good agent-assisted release tasks

Agent workflows are most valuable for tasks that are repetitive but still require repository awareness.

Examples:

- checking pending changesets before a release
- generating or reviewing changelog content
- running `pubm --preflight` and interpreting failures
- discovering external version references with `pubm sync --discover`
- scaffolding plugin or release configuration

## Recommended release pattern with an agent

The safest pattern is:

1. inspect the repository state
2. review pending changesets
3. run preview or dry-run commands
4. ask for approval before real publish

Typical command sequence:

```bash
pubm changesets status --verbose
pubm changesets version --dry-run
pubm patch --preview
pubm --preflight
```

This gives the agent enough context to explain:

- which packages will move
- what the resulting versions look like
- whether registry auth is ready
- which step is likely to fail

## Suggested slash-command ideas

If you are building an internal Claude Code workflow around `pubm`, these commands tend to have the highest leverage.

### `/publish-preview`

Runs:

```bash
pubm changesets status --verbose
pubm changesets version --dry-run
pubm patch --preview
```

Use when you want a release summary without making changes.

### `/publish-preflight`

Runs:

```bash
pubm --preflight
```

Use when you want CI-style validation before merging or tagging.

### `/version-sync`

Runs:

```bash
pubm sync --discover
```

Use when you need to find version strings in docs, READMEs, or sidecar manifests.

### `/changeset-check`

Runs:

```bash
pubm changesets status --verbose
```

Use when reviewing a pull request to confirm the release intent is recorded clearly.

## Safety rules worth encoding in a skill

If you build a Claude Code skill around `pubm`, the skill should strongly prefer:

- preview before publish
- explicit user approval before irreversible steps
- surfacing registry targets and versions in plain language
- stopping when Git state or tokens are not ready

Bad agent behavior:

- publishing just because a branch looks ready
- creating versions without showing the resulting plan
- swallowing a failed preflight and continuing

Good agent behavior:

- explain what is going to be published
- show the exact command that would run
- separate diagnosis from execution

## Current status

The `pubm` docs and codebase already support the building blocks an agent needs:

- explicit changesets workflow
- dry-run and preview commands
- preflight validation
- version-reference discovery
- plugin-defined commands

That means you can build productive Claude Code integrations today even if your repository-specific slash commands or marketplace packaging are still internal.

## Practical advice

- keep the skill prompt focused on repository inspection and release safety
- expose `pubm` commands directly rather than translating them into vague natural-language actions
- treat `pubm patch` as the last step, not the default step

For the underlying command surface, read the [CLI Reference](/pubm/reference/cli/).
