`pubm` is intentionally CLI-first, but real projects quickly outgrow flag-only configuration. `pubm.config.*` is where you keep repository-level publish targets, package layout, plugins, and changeset-related versioning rules.

## Config file discovery

`pubm` looks for the first matching file in the current working directory:

1. `pubm.config.ts`
2. `pubm.config.mts`
3. `pubm.config.cts`
4. `pubm.config.js`
5. `pubm.config.mjs`
6. `pubm.config.cjs`

TypeScript config files are supported directly. The loader bundles the file before evaluation, so you do not need a separate runtime transpilation step.

## Minimal config

For a package that has `package.json` and `jsr.json`, `pubm` infers the registries automatically. No `registries` field is needed:

```ts
import { defineConfig } from "@pubm/core";

export default defineConfig({});
```

## How registries are inferred

`pubm` inspects manifest files in each package directory and maps them to registries:

| Manifest | Inferred registry |
|---|---|
| `package.json` | `npm` |
| `jsr.json` | `jsr` |
| `Cargo.toml` | `crates` |

Two additional rules apply for npm:

- `publishConfig.registry` in `package.json` replaces `npm` with that custom URL
- `.npmrc` registry replaces `npm` when `publishConfig` is absent

## How config and CLI flags interact

The current runtime behavior is:

- config provides repository defaults for `packages` and `plugins`
- versioning-related config such as `versioning`, `fixed`, and `linked` is used by changeset flows
- CLI flags still control per-run publish behavior
- registries are inferred from manifests; `packages[].registries` overrides that inference

For a one-off publish to a specific registry:

```bash
pubm patch --registry npm --tag beta
```

## Common repository-level options

```ts
import { defineConfig } from "@pubm/core";
import { externalVersionSync } from "@pubm/plugin-external-version-sync";

export default defineConfig({
  plugins: [
    externalVersionSync({
      targets: [
        {
          file: "README.md",
          pattern: /pubm@([\\w.-]+)/,
        },
      ],
    }),
  ],
  snapshot: {
    useCalculatedVersion: false,
  },
});
```

These values cover the most common needs:

- `packages`: explicit multi-package publish layout (registries inferred from manifests unless overridden)
- `plugins`: release-time automation such as version sync or custom commands
- `snapshot`: customize snapshot version generation

## Multi-package configuration

For workspaces or mixed-language repositories, declare packages explicitly. Registries are inferred from each package's manifest files:

```ts
import { defineConfig } from "@pubm/core";

export default defineConfig({
  versioning: "independent",
  packages: [
    {
      path: "packages/core",
      // package.json + jsr.json → npm + jsr inferred automatically
    },
    {
      path: "packages/cli",
      registries: ["npm"], // override: publish only to npm even if jsr.json exists
    },
    {
      path: "crates/pubm-rs",
      // Cargo.toml → crates inferred automatically
    },
  ],
});
```

Use explicit package entries when:

- you want `pubm` to publish only a subset of discovered workspace packages
- JavaScript and Rust packages live in the same repository
- you need to override inferred registries or add private registries

To add a private registry alongside the inferred ones:

```ts
packages: [
  {
    path: "packages/a",
    registries: [
      "npm",
      { url: "https://npm.internal.com", token: { envVar: "INTERNAL_NPM_TOKEN" } },
    ],
  },
],
```

## Versioning strategies

### Independent versioning

Each package gets its own version and tag.

```ts
export default defineConfig({
  versioning: "independent",
});
```

Use this for library suites where packages evolve on different schedules.

### Fixed versioning

Every affected package shares one version.

```ts
export default defineConfig({
  versioning: "fixed",
});
```

Use this when the repository is consumed as one product and you want one release tag per publish.

### Fixed and linked groups

You can keep most packages independent while forcing selected packages to move together:

```ts
export default defineConfig({
  versioning: "independent",
  fixed: [["@acme/core", "@acme/react"]],
  linked: [["@acme/cli", "@acme/config"]],
});
```

- `fixed`: all packages in the group end up on the exact same version
- `linked`: the highest bump type propagates across the group

## Validation settings

Some typed config fields are part of the public schema but are not yet wired into every publish path. Prefer documenting operational behavior around the CLI flags and workflows you actively use in your repository, rather than assuming every typed field changes runtime behavior today.

## Snapshot settings

Snapshots are good for preview releases and branch-specific validation.

```ts
export default defineConfig({
  snapshot: {
    useCalculatedVersion: true,
    prereleaseTemplate: "{tag}-{timestamp}",
  },
});
```

Examples:

- `useCalculatedVersion: false` -> `0.4.1-snapshot-20260312...`
- `useCalculatedVersion: true` -> next calculated semver base with a snapshot suffix

## Plugins

Add plugins through the `plugins` array:

```ts
import { defineConfig } from "@pubm/core";
import { externalVersionSync } from "@pubm/plugin-external-version-sync";

export default defineConfig({
  plugins: [
    externalVersionSync({
      targets: [
        {
          file: "README.md",
          pattern: /pubm@([\\w.-]+)/,
        },
      ],
    }),
  ],
});
```

Read the [Plugins API Reference](/pubm/reference/plugins/) for hook order, command registration, and custom registry support.

## Working with optional dependencies in config

The config loader is designed to handle some optional imports without immediately failing. This is useful if your repository conditionally imports tooling that is only installed in specific packages.

Still, keep config files simple:

- avoid expensive runtime work during module evaluation
- keep imports deterministic
- move project logic into small helper modules if the config file starts to become a program

## Example patterns

### Simple library

If the package has `package.json` and `jsr.json`, registries are inferred automatically:

```ts
import { defineConfig } from "@pubm/core";

export default defineConfig({});
```

### App monorepo

Packages with only `package.json` resolve to `npm`. No explicit `registries` needed unless you want to override.

```ts
import { defineConfig } from "@pubm/core";

export default defineConfig({
  versioning: "fixed",
  packages: [
    { path: "packages/web" },
    { path: "packages/server" },
  ],
});
```

### Mixed JS + Rust workspace

```ts
import { defineConfig } from "@pubm/core";

export default defineConfig({
  versioning: "independent",
  packages: [
    { path: "packages/sdk" },
    { path: "crates/sdk-core" },
  ],
});
```

### Private registry

```ts
import { defineConfig } from "@pubm/core";

export default defineConfig({
  packages: [
    {
      path: "packages/a",
      registries: [
        "npm",
        { url: "https://npm.internal.com", token: { envVar: "INTERNAL_NPM_TOKEN" } },
      ],
    },
  ],
});
```
