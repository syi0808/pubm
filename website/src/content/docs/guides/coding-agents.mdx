`pubm` is a good fit for coding-agent workflows because the release path is explicit: inspect state, preview the plan, validate credentials, then publish. That makes it much easier to encode safe agent behavior without guessing.

## What this guide covers

This page describes the recommended integration model:

- expose release workflows as explicit CLI commands, plugin commands, or skills
- keep preview and validation steps first-class
- bundle setup automation with the same project that owns the release pipeline
- mirror that bundle into agent-specific install surfaces when needed

## Good agent-assisted release tasks

Agent workflows are most valuable for tasks that are repetitive but still require repository awareness.

Examples:

- checking pending changesets before a release
- generating or reviewing changelog content
- running `pubm --preflight` and interpreting failures
- discovering external version references with `pubm sync --discover`
- scaffolding plugin or release configuration

## The `pubm plugin` bundle

This repository already includes a coding-agent bundle under:

```text
plugins/pubm-plugin/skills/
```

The current skill set covers:

- `publish-setup`: install/configure `pubm`, registries, CI, and changesets
- `publish-preview`: run preview-first release inspection
- `publish`: guarded publish workflow with explicit confirmation
- `version-sync`: find non-manifest version references
- `create-plugin`: scaffold a new `pubm` plugin package

This is the pattern to copy when you want agent support to be a first-class part of your release tooling:

- keep the runtime plugin or CLI command in the repository
- keep the skill prompts and safety rails in the repository
- version both together

## Recommended release pattern with an agent

The safest operating pattern is:

1. inspect the repository state
2. review pending changesets
3. run preview or dry-run commands
4. ask for approval before real publish

Typical command sequence:

```bash
pubm changesets status --verbose
pubm changesets version --dry-run
pubm patch --preview
pubm --preflight
```

This gives the agent enough context to explain:

- which packages will move
- what the resulting versions look like
- whether registry auth is ready
- which step is likely to fail

## Suggested command surfaces

If you are building an internal coding-agent workflow around `pubm`, these commands tend to have the highest leverage.

### `/publish-preview`

Runs:

```bash
pubm changesets status --verbose
pubm changesets version --dry-run
pubm patch --preview
```

Use when you want a release summary without making changes.

### `/publish-preflight`

Runs:

```bash
pubm --preflight
```

Use when you want CI-style validation before merging or tagging.

### `/version-sync`

Runs:

```bash
pubm sync --discover
```

Use when you need to find version strings in docs, READMEs, or sidecar manifests.

### `/changeset-check`

Runs:

```bash
pubm changesets status --verbose
```

Use when reviewing a pull request to confirm the release intent is recorded clearly.

## Skills plus plugin commands

The best integrations usually combine both:

- skills or prompts for sequencing, approvals, and repository inspection
- `pubm` plugin commands for deterministic setup or organization-specific actions

Examples:

- `pubm brew init` for Homebrew scaffolding
- a plugin command that bootstraps release metadata
- a repo-local skill that decides when to call `pubm --preflight`

## Agent-specific packaging

Different coding agents expose different install surfaces, but the packaging model should stay consistent.

### Agents with repo-local skills

Point the agent at the checked-in skill bundle and keep the prompts next to the code they operate on.

### Agents with slash-command systems

Map the high-signal flows directly:

- preview
- preflight
- setup
- version sync
- plugin scaffolding

### Agents with marketplaces or extension registries

Publish a thin wrapper that installs or exposes the same bundle. The marketplace package should describe:

- which commands it runs
- which files it may create or edit
- which tokens or secrets it expects
- which preview and approval steps are mandatory
- how it stays version-aligned with the repository plugin/skill bundle

Do not create a separate agent-only workflow that diverges from the repository's actual release process.

## Safety rules worth encoding

If you build a coding-agent skill or marketplace package around `pubm`, it should strongly prefer:

- preview before publish
- explicit user approval before irreversible steps
- surfacing registry targets and versions in plain language
- stopping when Git state or tokens are not ready

Bad agent behavior:

- publishing just because a branch looks ready
- creating versions without showing the resulting plan
- swallowing a failed preflight and continuing

Good agent behavior:

- explain what is going to be published
- show the exact command that would run
- separate diagnosis from execution

## What to ship in a marketplace listing

If an agent ecosystem has a marketplace, publish the smallest installable unit that still contains the full workflow context:

- the repo-local skill or prompt files
- any plugin command names the skill depends on
- the expected install location or activation step
- a short safety policy
- links to preview and troubleshooting commands

That keeps the marketplace listing honest: it is a distribution mechanism, not a second source of truth.

## Current status

The `pubm` codebase already provides the main primitives a coding agent needs:

- explicit changesets workflow
- dry-run and preview commands
- preflight validation
- version-reference discovery
- plugin-defined commands
- a checked-in `pubm plugin` skill bundle

That means you can build productive coding-agent integrations today even if the final packaging differs between agents.

## Practical advice

- keep the skill prompt focused on repository inspection and release safety
- expose `pubm` commands directly rather than translating them into vague natural-language actions
- treat `pubm patch` as the last step, not the default step
- keep setup automation in the same bundle as publish automation
- if you publish to an agent marketplace, mirror the repo bundle instead of rewriting it there

For the underlying command surface, read the [CLI Reference](/pubm/reference/cli/).
