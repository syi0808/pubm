This guide is for teams that already use coding agents such as Claude Code or Codex and want to add `pubm` release workflows into that existing setup.

The goal is not to introduce a new agent. The goal is to make your current agent understand how this repository wants release work to happen.

## What the `pubm plugin` gives your agent

This repository already ships an agent-facing bundle under `plugins/pubm-plugin/`.

Today that bundle contains:

- a Claude Code plugin manifest at `plugins/pubm-plugin/.claude-plugin/plugin.json`
- a checked-in skill bundle under `plugins/pubm-plugin/skills/`

That means the repo already has a concrete integration surface for agent-driven release work. Instead of writing ad hoc prompts like "set up publishing" or "release the project", you can install or adapt the `pubm plugin` bundle and keep the release workflow versioned with the codebase.

## Included skills

The current bundle focuses on skills rather than custom commands or subagents.

### `publish-setup`

Source: `plugins/pubm-plugin/skills/publish-setup/SKILL.md`

Use this first when wiring `pubm` into a repository through an agent. It is the entry point for installation, config generation, CI wiring, changesets, and optional version-sync setup.

### `publish-preview`

Source: `plugins/pubm-plugin/skills/publish-preview/SKILL.md`

Use this when the agent should explain the release plan before it mutates git state or publishes anything.

### `publish`

Source: `plugins/pubm-plugin/skills/publish/SKILL.md`

This is the guarded publish path. It is preview-first and requires confirmation before the real publish command.

### `version-sync`

Source: `plugins/pubm-plugin/skills/version-sync/SKILL.md`

Use this to find and wire version references outside package manifests such as docs snippets, metadata files, and sidecar config.

### `create-plugin`

Source: `plugins/pubm-plugin/skills/create-plugin/SKILL.md`

Use this when the agent needs to scaffold a new `pubm` plugin package with hooks, commands, and tests.

## Integration model

If you already have a preferred coding agent, keep the split like this:

- keep runtime release behavior in `pubm` itself or in a `pubm` plugin
- keep agent workflow and approval policy in checked-in skills
- keep the bundle inside the repository that owns the release process

That gives your agent one source of truth for:

- which setup steps are allowed
- which files may be generated
- which preview and approval gates are mandatory
- which repository conventions the agent must follow

## Start with setup, not publish

If you are integrating `pubm` into an agent-enabled repository for the first time, start with `publish-setup`.

That skill is the bridge between the raw CLI and the repository's actual release conventions. It handles the setup tasks teams usually forget to encode:

- registry selection
- CI wiring
- changesets workflow setup
- `.gitignore` behavior for `.pubm/`
- optional version-sync configuration

That is why the [Quick Start](/pubm/guides/quick-start/) guide should point existing agent users to setup first and not straight to `pubm patch`.

## Skills and command surface

Skills should not replace the CLI. They should orchestrate it.

For `pubm`, the practical rule is:

- use `pubm` commands for deterministic release operations
- use skills for setup sequencing, project inspection, safety checks, and approvals

Typical pairings:

- `publish-setup` inspects the repo and writes the right `pubm` config
- `publish-preview` runs preview-oriented commands before publish
- `version-sync` runs `pubm sync --discover` and then wires selected targets into config
- `create-plugin` scaffolds a new `pubm` plugin package when you need repo-specific extensions

## Claude Code installation

Claude Code's official plugin model is marketplace-based: add a marketplace, install a plugin, then choose its installation scope. See Anthropic's official guides for [discovering plugins](https://code.claude.com/docs/en/discover-plugins) and [creating plugins](https://code.claude.com/docs/en/plugins).

Anthropic's official install flow is:

1. add the marketplace with `/plugin marketplace add ...`
2. install the plugin with `/plugin install plugin-name@marketplace-name`
3. choose the scope in the interactive UI when needed

Examples from the Claude Code docs:

```text
/plugin marketplace add your-org/claude-plugins
/plugin install plugin-name@marketplace-name
```

Installation scopes matter:

- User scope: install for yourself across projects
- Project scope: share the plugin through the repository and write it into `.claude/settings.json`
- Local scope: install only for yourself in the current repository

When `pubm` is distributed through a marketplace, this is the installation model Claude Code users should follow.

## Safety rules to encode

Any coding-agent integration around `pubm` should strongly prefer:

- setup before publish in new repositories
- preview before publish in existing repositories
- explicit approval before irreversible actions
- plain-language summaries of versions, registries, and side effects
- stopping when git state, auth, or CI assumptions are not ready

Bad agent behavior:

- publishing because the branch "looks ready"
- generating config files without showing the intended result
- swallowing failed preflight output and continuing anyway
- building a custom release path that skips the repository's checked-in setup skill

Good agent behavior:

- starts with `publish-setup` when the repo is not wired yet
- shows the exact preview command before asking to publish
- explains the likely failure point when validation fails
- keeps the repository bundle authoritative

## Practical advice

- treat `pubm plugin` as the integration layer for agents you already use
- start with `publish-setup`, not `publish`
- expose `pubm` commands directly rather than translating them into vague natural-language actions
- keep setup automation and publish automation in the same checked-in bundle
- use Claude Code's official marketplace install flow when a marketplace is available
- treat `pubm patch` as the last step, not the default step

For the underlying command surface, read the [CLI Reference](/pubm/reference/cli/).
