Use this guide to choose how a real release moves from local checks to published packages.

`pubm` currently supports three release workflows:

| Workflow              | Commands or actions                                          | Use when                                                                            |
| --------------------- | ------------------------------------------------------------ | ----------------------------------------------------------------------------------- |
| Direct Release        | `pubm`                                                       | One trusted local terminal or controlled job should run the whole release.          |
| Split CI Release      | Local `pubm --phase prepare`, then CI `pubm --phase publish` | Local preparation should write versions and tags; CI should publish from tokens.    |
| GitHub Release PR     | `pubm workflow install github`                               | Release file changes should be reviewed in PRs before CI publishes merged releases. |

## Direct Release

Direct Release runs the full release pipeline in one command:

```bash
pubm
```

Use it when the person or job starting the release can safely hold registry credentials, push the release commit and tags, and publish packages.

Direct Release:

- validates the project and Git state
- runs prerequisite and required-condition checks
- runs tests and build unless skipped
- consumes pending changesets when present
- writes manifest versions
- creates the release commit and tags
- publishes packages to the selected registries
- creates GitHub Releases when enabled
- rolls back publish-side failures where possible

## Split CI Release

Split CI Release separates the mutable preparation step from the non-interactive publish step:

```bash
pubm --phase prepare
```

```bash
pubm --phase publish
```

Use it when maintainers should approve or prepare the release locally, but registry publishing should happen in CI with repository secrets.

### Prepare for CI publish

Run Prepare for CI publish locally:

```bash
pubm --phase prepare
```

Prepare for CI publish:

- collects registry tokens and plugin credentials
- optionally syncs credentials to GitHub Secrets
- runs prerequisite and required-condition checks
- runs tests and build unless skipped
- consumes pending changesets during versioning when present
- writes manifest versions
- creates the release commit and tags
- pushes the release commit and tags for CI to pick up
- dry-runs publish tasks instead of performing the real publish
- does not publish packages

### Publish prepared release

Run Publish prepared release in CI:

```bash
pubm --phase publish
```

Publish prepared release:

- reads the versions already written to package manifests
- publishes packages to the selected registries
- creates GitHub Releases and uploads release assets when configured
- expects registry and plugin tokens from the CI environment
- runs without interactive prompts
- does not redo versioning
- does not create tags again
- does not push the release commit or tags again

## GitHub Release PR

GitHub Release PR uses split `pubm-actions` sub-actions:

```bash
pubm workflow install github
```

The `release-pr` action reads pending changesets or conventional commits, writes manifest versions, changelogs, consumed changesets, and `afterVersion` hook output into a managed PR. The `publish` action runs only after a labeled release PR is merged, then creates tags on the merged commit, publishes the matching package scope, creates GitHub Releases, and runs publish-side hooks.

Use it when branch protection requires PR approval for version/changelog changes or when independent monorepo packages should get separate release PRs.

Release PRs are rolling pending PRs per release scope. If another releasable change lands before the PR is merged, pubm updates the existing PR for that package or linked/fixed group instead of opening a duplicate for the new version. The default release PR branch template is versionless (`pubm/release/{scopeSlug}`); the current version is shown in the PR title and body.

## Current scope

Fully automated CI, where CI decides and prepares the release without local `pubm --phase prepare`, is not a supported current workflow.
