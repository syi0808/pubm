This reference describes the typed shape of `pubm.config.*` as exposed by `defineConfig()` from `@pubm/core`.

The config schema is broader than the current runtime wiring. Today, the most important fields in active publish flows are:

- `packages`
- `plugins`
- `versioning`, `fixed`, and `linked` for changeset/version planning
- `snapshot.useCalculatedVersion` for snapshot releases

```ts
import { defineConfig } from "@pubm/core";

export default defineConfig({
  // options
});
```

## Default config

These are the defaults resolved by the config layer:

```ts
export default defineConfig({
  versioning: "independent",
  branch: "main",
  changelog: true,
  changelogFormat: "default",
  commit: false,
  access: "public",
  fixed: [],
  linked: [],
  updateInternalDependencies: "patch",
  ignore: [],
  tag: "latest",
  contents: ".",
  saveToken: true,
  releaseDraft: true,
  releaseNotes: true,
  rollbackStrategy: "individual",
  packages: [{ path: "." }],
  validate: {
    cleanInstall: true,
    entryPoints: true,
    extraneousFiles: true,
  },
  snapshot: {
    useCalculatedVersion: false,
    prereleaseTemplate: "{tag}-{timestamp}",
  },
});
```

## Top-level options

## `versioning`

- Type: `"independent" | "fixed"`
- Default: `"independent"`

Controls how versions are assigned in multi-package repositories.

## `branch`

- Type: `string`
- Default: `"main"`

Branch that must match `HEAD` during guarded release runs.
Typed config field for release-branch policy. If you need branch enforcement today, prefer the CLI `--branch` flag.

## `packages`

- Type: `PackageConfig[]`
- Default: `[{ path: "." }]`

Explicitly declares publishable packages. Registries are inferred from manifest files unless overridden per package.

```ts
packages: [
  { path: "packages/core" },
  { path: "crates/pubm-rs" },
];
```

Use this when package discovery alone is not expressive enough.

## `changelog`

- Type: `boolean | string`
- Default: `true`

Controls changelog generation configuration. `true` enables default behavior; a string can be used as a custom path convention in config consumers.

## `changelogFormat`

- Type: `"default" | "github" | string`
- Default: `"default"`

Changelog formatting strategy exposed by the config API.

## `commit`

- Type: `boolean`
- Default: `false`

Commit-related config flag exposed by the API.

## `access`

- Type: `"public" | "restricted"`
- Default: `"public"`

Relevant for npm-style publishing where package access level matters.

## `fixed`

- Type: `string[][]`
- Default: `[]`

Groups of packages that must always resolve to the same version.

## `linked`

- Type: `string[][]`
- Default: `[]`

Groups of packages that should share the highest bump type without necessarily sharing the same exact version at all times.

## `updateInternalDependencies`

- Type: `"patch" | "minor"`
- Default: `"patch"`

Declares the workspace internal dependency update strategy.

## `ignore`

- Type: `string[]`
- Default: `[]`

Package names or paths to exclude from workspace discovery.

## `tag`

- Type: `string`
- Default: `"latest"`

Default distribution tag for publish runs.
Typed config field for distribution tags. Per-run tag selection is currently most reliable through the CLI `--tag` flag.

## `contents`

- Type: `string`
- Default: `"."`

Directory to `chdir` into before running the publish pipeline.
Typed config field for alternate publish roots. Use the CLI `--contents` flag for the current runtime publish path.

## `saveToken`

- Type: `boolean`
- Default: `true`

Whether supported registry tokens should be stored for reuse on local machines.
Typed config field for token persistence policy. Operationally, `--no-save-token` is the flag that affects a publish run today.

## `releaseDraft`

- Type: `boolean`
- Default: `true`

Whether the normal publish flow should open the GitHub release draft step.
Typed config field for release-draft policy. Use the CLI `--no-release-draft` flag to control the current publish path.

## `releaseNotes`

- Type: `boolean`
- Default: `true`

Release-notes related config flag exposed by the API.

## `rollbackStrategy`

- Type: `"individual" | "all"`
- Default: `"individual"`

Rollback policy field exposed by the config API for multi-target release handling.

## `plugins`

- Type: `PubmPlugin[]`
- Default: `[]`

Registers plugins that can add hooks, commands, registries, or ecosystems.

## `validate`

- Type: `ValidateConfig`
- Default:

```ts
{
  cleanInstall: true,
  entryPoints: true,
  extraneousFiles: true,
}
```

Typed validation config surface. These fields are part of the public schema, but they are not yet active toggles in the current publish pipeline.

### `validate.cleanInstall`

- Type: `boolean`
- Default: `true`

Reserved validation toggle.

### `validate.entryPoints`

- Type: `boolean`
- Default: `true`

Reserved validation toggle.

### `validate.extraneousFiles`

- Type: `boolean`
- Default: `true`

Reserved validation toggle.

## `snapshot`

- Type: `SnapshotConfig`
- Default:

```ts
{
  useCalculatedVersion: false,
  prereleaseTemplate: "{tag}-{timestamp}",
}
```

Controls snapshot version generation.

### `snapshot.useCalculatedVersion`

- Type: `boolean`
- Default: `false`

Whether the snapshot base should use the next calculated semver instead of the current package version.

### `snapshot.prereleaseTemplate`

- Type: `string`
- Default: `"{tag}-{timestamp}"`

Template used to build the snapshot suffix.

## `PackageConfig`

Package-level entries are shaped like this:

```ts
interface PackageConfig {
  path: string;
  registries?: Array<"npm" | "jsr" | "crates" | PrivateRegistryConfig>;
  ecosystem?: "js" | "rust";
  buildCommand?: string;
  testCommand?: string;
}

interface PrivateRegistryConfig {
  url: string;
  token: { envVar: string } | { secureStore: string };
}
```

### `path`

- Type: `string`
- Required: yes

Relative path to the package root.

### `registries`

- Type: `Array<"npm" | "jsr" | "crates" | PrivateRegistryConfig>`
- Required: no

Override the inferred registry targets for that package. When omitted, `pubm` infers registries from manifest files:

- `package.json` â†’ `npm`
- `jsr.json` â†’ `jsr`
- `Cargo.toml` â†’ `crates`
- `publishConfig.registry` in `package.json` â†’ replaces `npm` with that registry URL
- `.npmrc` registry â†’ replaces `npm` when `publishConfig` is absent

Use this field when inference is insufficient or when you need to add private registries alongside built-in ones:

```ts
packages: [
  {
    path: "packages/a",
    registries: [
      "npm",
      { url: "https://npm.internal.com", token: { envVar: "INTERNAL_NPM_TOKEN" } },
    ],
  },
];
```

### `ecosystem`

- Type: `"js" | "rust"`
- Required: no

Use this when the package type cannot be inferred reliably from the path and manifest files alone.

### `buildCommand`

- Type: `string`
- Required: no

Per-package build command field exposed by the config type.

### `testCommand`

- Type: `string`
- Required: no

Per-package test command field exposed by the config type.

## `ValidateConfig`

## Examples

## Single package â€” inferred registries

A package with both `package.json` and `jsr.json` needs no explicit registry config. `pubm` infers `npm` and `jsr` from the manifests.

```ts
import { defineConfig } from "@pubm/core";

export default defineConfig({});
```

## Fixed-version monorepo

Registries are inferred per package from each package's manifest files. Override with `registries` only when inference is wrong.

```ts
import { defineConfig } from "@pubm/core";

export default defineConfig({
  versioning: "fixed",
  packages: [
    { path: "packages/web" },
    { path: "packages/server" },
  ],
});
```

## Mixed ecosystem repository

```ts
import { defineConfig } from "@pubm/core";

export default defineConfig({
  versioning: "independent",
  packages: [
    { path: "packages/sdk" },
    { path: "crates/sdk-core" },
  ],
  snapshot: {
    useCalculatedVersion: true,
  },
});
```

## Private registry alongside npm

```ts
import { defineConfig } from "@pubm/core";

export default defineConfig({
  packages: [
    {
      path: "packages/a",
      registries: [
        "npm",
        { url: "https://npm.internal.com", token: { envVar: "INTERNAL_NPM_TOKEN" } },
      ],
    },
  ],
});
```

## Practical notes

- Prefer config for stable repository defaults and CLI flags for temporary overrides.
- Rely on manifest-based inference where possible; use `packages[].registries` only when the inferred targets are wrong or when private registries need to be added.
- Keep config files lightweight; avoid running expensive logic during module evaluation.

For usage patterns and config layouts, read the [Configuration](/pubm/guides/configuration/) guide.
