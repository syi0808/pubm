The plugin system lets you extend `pubm` without forking the core publish pipeline.

## `PubmPlugin`

```ts
export interface PubmPlugin {
  name: string;
  registries?: PackageRegistry[];
  ecosystems?: Ecosystem[];
  hooks?: PluginHooks;
  commands?: PluginCommand[];
}
```

## Hooks

```ts
export interface PluginHooks {
  // Lifecycle hooks
  beforeTest?: HookFn;
  afterTest?: HookFn;
  beforeBuild?: HookFn;
  afterBuild?: HookFn;
  beforeVersion?: HookFn;
  afterVersion?: HookFn;
  beforePublish?: HookFn;
  afterPublish?: HookFn;
  beforePush?: HookFn;
  afterPush?: HookFn;
  afterRelease?: AfterReleaseHookFn;
  onError?: ErrorHookFn;
  onRollback?: HookFn;
  onSuccess?: HookFn;

  // Asset pipeline hooks
  resolveAssets?: AssetPipelineHooks["resolveAssets"];
  transformAsset?: AssetPipelineHooks["transformAsset"];
  compressAsset?: AssetPipelineHooks["compressAsset"];
  nameAsset?: AssetPipelineHooks["nameAsset"];
  generateChecksums?: AssetPipelineHooks["generateChecksums"];
  uploadAssets?: AssetPipelineHooks["uploadAssets"];
}
```

### Asset pipeline hooks

Six hooks let plugins intercept every stage of the release asset pipeline. They run between the build step and the GitHub Release upload.

| Hook | When it runs | Input ã Output |
|------|-------------|----------------|
| `resolveAssets` | After glob matching | `ResolvedAsset[]` ã `ResolvedAsset[]` |
| `transformAsset` | Per asset, before compression | `ResolvedAsset` ã `TransformedAsset \| TransformedAsset[]` |
| `compressAsset` | Per asset, replaces built-in compression | `TransformedAsset` ã `CompressedAsset` |
| `nameAsset` | Per asset, after compression | `CompressedAsset` ã `string` |
| `generateChecksums` | After all assets are named and hashed | `PreparedAsset[]` ã `PreparedAsset[]` |
| `uploadAssets` | After GitHub Release upload | `PreparedAsset[]` ã `UploadedAsset[]` |

When multiple plugins register the same asset pipeline hook, they are chained in registration order. `uploadAssets` is an exception: each plugin receives the same input independently and all results are concatenated.

Read [Asset Pipeline Hooks](/pubm/guides/asset-pipeline-hooks/) for full signatures, examples, and composition rules.

## Hook signatures

```ts
type HookFn = (ctx: PubmContext) => Promise<void> | void;
type ErrorHookFn = (ctx: PubmContext, error: Error) => Promise<void> | void;
type AfterReleaseHookFn = (
  ctx: PubmContext,
  releaseCtx: ReleaseContext,
) => Promise<void> | void;
```

## `PubmContext`

Hooks receive the current pipeline context:

```ts
export interface PubmContext {
  readonly config: ResolvedPubmConfig;
  readonly options: ResolvedOptions;
  readonly cwd: string;
  runtime: {
    tag: string;
    promptEnabled: boolean;
    cleanWorkingTree: boolean;
    pluginRunner: PluginRunner;
    versionPlan?: VersionPlan;
    releaseContext?: ReleaseContext;
    tempDir?: string;
  };
}
```

Key fields:

- `ctx.config` ã resolved pubm config (packages, registries, plugins, releaseAssets, etc.)
- `ctx.options` ã resolved CLI options (version, tag, preview, ci, etc.)
- `ctx.cwd` ã working directory
- `ctx.runtime` ã mutable runtime state (version plan, temp directory, etc.)

## Example

```ts
const notifyPlugin = {
  name: "notify-plugin",
  hooks: {
    async afterPublish(ctx) {
      const label =
        ctx.versions && ctx.versions.size > 0
          ? [...ctx.versions].map(([name, version]) => `${name}@${version}`).join(", ")
          : `v${ctx.version}`;

      console.log(`Published ${label}`);
    },
  },
};
```

## Custom commands

```ts
export interface PluginCommand {
  name: string;
  description: string;
  subcommands?: PluginSubcommand[];
}

export interface PluginSubcommand {
  name: string;
  description: string;
  options?: PluginCommandOption[];
  action: (args: Record<string, unknown>) => Promise<void>;
}
```

## Custom registries and ecosystems

Use custom registries when you need another publish destination. Use custom ecosystems when you need to teach `pubm` how to read and write versions for another package type.
