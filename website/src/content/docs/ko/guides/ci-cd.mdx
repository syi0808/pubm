`pubm`은 로컬 터미널과 CI 러너 모두에서 일관되게 동작하도록 설계되었습니다. 이 문서는 로컬 검증, CI 필수 입력, 안정적인 릴리스 파이프라인 구성을 설명합니다.

## 권장 릴리스 모델

일반적으로 두 단계가 가장 안전합니다.

1. 로컬 또는 제어된 잡에서 버전과 changelog 준비
2. CI에서 태그와 비대화형 자격 증명으로 실제 publish

```bash
pubm changesets version
pubm --preflight
pubm patch --preview
```

```bash
pubm --ci
```

## CI 활성화 전에 preflight 실행

```bash
pubm --preflight
```

preflight는 다음을 수행합니다.

- 레지스트리 토큰 수집
- 필요 시 GitHub Secrets 동기화
- 프롬프트 비활성화 후 CI처럼 동작
- prerequisite / condition 검사
- 설정된 레지스트리에 대해 dry-run publish

## 필수 환경 변수

| 변수 | 레지스트리 |
|------|------------|
| `NODE_AUTH_TOKEN` | npm |
| `JSR_TOKEN` | jsr |
| `CARGO_REGISTRY_TOKEN` | crates.io |

원칙:

- 자동화 전용 토큰 사용
- 가능한 최소 권한 사용
- CI에서 프롬프트에 의존하지 않기

## GitHub Secrets 동기화

```bash
pubm secrets sync
pubm secrets sync --registry npm,jsr
```

`gh` CLI가 설치 및 인증되어 있어야 합니다.

## GitHub Actions 예시

```yaml
name: Release

on:
  push:
    tags:
      - "v*"
      - "@*"

jobs:
  publish:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 24
      - run: pnpm install --frozen-lockfile
      - run: pubm --ci
        env:
          NODE_AUTH_TOKEN: ${{ secrets.NODE_AUTH_TOKEN }}
          JSR_TOKEN: ${{ secrets.JSR_TOKEN }}
          CARGO_REGISTRY_TOKEN: ${{ secrets.CARGO_REGISTRY_TOKEN }}
```

## 자주 겪는 문제

### shallow checkout

태그/히스토리 기반 동작에는 `fetch-depth: 0`이 필요합니다.

### 토큰 누락

CI에서는 프롬프트가 없으므로 토큰이 없으면 즉시 실패합니다.

### npm 2FA

헤드리스 환경에서는 일회용 OTP 흐름이 동작하지 않으므로 automation token을 사용하세요.

### `contents` 결과물이 없음

`dist/`에서 publish한다면 build가 먼저 해당 디렉터리를 생성해야 합니다.

## 운영 권장사항

- 새 레지스트리를 추가할 때마다 `pubm --preflight` 실행
- PR 테스트 잡과 릴리스 잡 분리
- 브랜치 기반보다 태그 기반 publish 선호
- 초반 몇 번의 릴리스는 수동 검토 유지
