`pubm`은 로컬 터미널과 CI 러너 모두에서 일관되게 동작하도록 설계되었습니다. 핵심은 일반 `pubm` 파이프라인이 버전 단계에서 pending changeset을 직접 소비한다는 점입니다. 그래서 CI 준비를 위해 `pubm changesets version`을 별도 단계로 둘 필요가 없습니다.

## 권장 릴리스 모델

두 단계로 운영하는 것이 가장 단순합니다.

1. 로컬 또는 제어된 릴리스 잡에서 `pubm --preflight`
2. CI에서 `pubm --ci`

```bash
pubm --preflight
```

```bash
pubm --ci
```

## preflight가 하는 일

`pubm --preflight`는 단순 토큰 점검이 아닙니다. 실제 publish 직전까지 릴리스 파이프라인을 실행합니다.

- 레지스트리 토큰 수집
- 필요 시 GitHub Secrets 동기화
- prerequisite / required-condition 검사
- tests / build 실행
- pending changeset이 있으면 버전 단계에서 소비
- 릴리스 커밋 생성 및 git push 수행
- 실제 publish 대신 dry-run publish 태스크 실행

그래서 preflight 이후에 `pubm patch --preview`를 다시 실행할 필요가 없습니다.

## 필수 환경 변수

| 변수 | 레지스트리 |
|------|------------|
| `NODE_AUTH_TOKEN` | npm |
| `JSR_TOKEN` | jsr |
| `CARGO_REGISTRY_TOKEN` | crates.io |

## GitHub Secrets 동기화

```bash
pubm secrets sync
pubm secrets sync --registry npm,jsr
```

## GitHub Actions 예시

```yaml
name: Release

on:
  push:
    tags:
      - "v*"
      - "@*"

jobs:
  publish:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 24
      - run: pnpm install --frozen-lockfile
      - run: pubm --ci
        env:
          NODE_AUTH_TOKEN: ${{ secrets.NODE_AUTH_TOKEN }}
          JSR_TOKEN: ${{ secrets.JSR_TOKEN }}
          CARGO_REGISTRY_TOKEN: ${{ secrets.CARGO_REGISTRY_TOKEN }}
```

## 자주 겪는 문제

### shallow checkout

태그/히스토리 기반 동작에는 `fetch-depth: 0`이 필요합니다.

### 토큰 누락

CI에서는 프롬프트가 없으므로 토큰이 없으면 즉시 실패합니다.

### npm 2FA

헤드리스 환경에서는 automation token을 사용하세요.

### `contents` 결과물이 없음

`dist/`에서 publish한다면 build가 먼저 해당 디렉터리를 생성해야 합니다.
