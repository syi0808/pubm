이 문서는 이미 코딩 에이전트를 사용하는 팀이 기존 워크플로에 `pubm` 릴리스 흐름을 붙이는 방법을 설명합니다.

## 가장 먼저 setup 자동화

저장소가 아직 연결되지 않았다면, 가장 중요한 에이전트 작업은 publish가 아니라 setup입니다.

먼저 `publish-setup`을 사용하세요. 이 스킬은 다음을 담당합니다.

- `pubm plugin` 번들 설치
- `pubm` 설정 생성 또는 갱신
- CI 연결
- changesets 활성화
- 필요하면 version-sync 대상 연결

에이전트가 번들을 가져오고 체크인된 setup 흐름을 따를 수 있도록 [INSTALLATION.md](/pubm/plugins/pubm-plugin/INSTALLATION.md) 링크를 제공하세요.

## `pubm plugin`이 제공하는 것

저장소에는 `plugins/pubm-plugin/` 아래 에이전트용 번들이 있습니다.

- `plugins/pubm-plugin/.claude-plugin/plugin.json`
- `plugins/pubm-plugin/skills/`

## 포함된 스킬

### `publish-setup`

에이전트로 저장소에 `pubm`을 처음 연결할 때 가장 먼저 사용합니다.

### `publish-preview`

git 상태를 바꾸거나 publish하기 전에 릴리스 계획을 설명해야 할 때 사용합니다.

### `publish`

실제 publish 경로입니다. preview를 먼저 수행하고, 실제 publish 전에 확인을 요구합니다.

### `version-sync`

문서 스니펫이나 메타데이터 파일처럼 패키지 매니페스트 밖의 버전 참조를 찾고 연결할 때 사용합니다.

### `create-plugin`

새로운 `pubm` 플러그인 패키지와 훅, 명령, 테스트를 스캐폴딩합니다.

## Claude Code 설치

Claude Code는 marketplace 기반 플러그인 설치 모델을 사용합니다.

일반적인 흐름:

1. marketplace 추가
2. 해당 marketplace에서 plugin 설치
3. Claude Code에서 설치 범위 선택

```text
/plugin marketplace add your-org/claude-plugins
/plugin install plugin-name@marketplace-name
```

marketplace를 사용하지 않는다면 대신 [INSTALLATION.md](/pubm/plugins/pubm-plugin/INSTALLATION.md) 링크를 에이전트에 제공하세요.

## 통합 원칙

- 실제 릴리스 동작은 `pubm` 본체나 `pubm` 플러그인에 둡니다.
- 에이전트 승인 정책과 작업 순서는 체크인된 skill에 둡니다.
- setup 자동화를 저장소 온보딩의 기준으로 둡니다.

## 스킬과 CLI의 역할 분리

- 결정적인 릴리스 작업은 `pubm` 명령으로 수행
- 스킬은 설정 순서, 프로젝트 점검, 안전 장치, 승인 흐름을 오케스트레이션

## 에이전트에 반드시 넣어야 할 안전 규칙

- 새 저장소는 publish보다 setup 우선
- 기존 저장소도 publish 전에 preview 우선
- 되돌릴 수 없는 작업 전에는 명시적 승인
- 버전, 레지스트리, 부작용을 평문으로 설명
- git 상태, 인증, CI 가정이 준비되지 않았으면 중단

## 실무 팁

- `publish-setup`을 가장 중요한 통합 표면으로 취급하세요.
- setup 자동화와 publish 자동화를 같은 체크인 번들 안에 두세요.
- `pubm` 명령을 직접 노출하고, 모호한 자연어 액션으로 감추지 마세요.
- `pubm patch`는 기본 단계가 아니라 마지막 단계로 취급하세요.
