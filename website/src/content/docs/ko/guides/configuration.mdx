`pubm`은 의도적으로 CLI 우선 도구지만, 실제 프로젝트는 금방 플래그만으로는 부족해집니다. `pubm.config.*`는 저장소 수준의 publish 대상, 패키지 레이아웃, 플러그인, changeset 기반 버전 규칙을 두는 곳입니다.

`pubm init`을 실행해 이 설정 파일을 대화형으로 생성합니다. 마법사는 선택 사항이 아래 표시된 기본값과 다를 때만 설정 파일을 생성합니다.

## 지원되는 설정 파일

`pubm`은 다음 파일 이름을 지원합니다.

1. `pubm.config.ts`
2. `pubm.config.mts`
3. `pubm.config.cts`
4. `pubm.config.js`
5. `pubm.config.mjs`
6. `pubm.config.cjs`

TypeScript 설정 파일도 직접 지원합니다. 로더가 평가 전에 파일을 번들링하므로, 별도의 런타임 트랜스파일 단계는 필요하지 않습니다.

## 기본 설정

These are the default user-facing settings.

```ts
export default defineConfig({
  branch: "main",
  commit: false,
  access: "public",
  release: {
    versioning: {
      mode: "independent",
      fixed: [],
      linked: [],
      updateInternalDependencies: "patch",
    },
    changesets: {
      directory: ".pubm/changesets",
    },
    commits: {
      format: "conventional",
      types: {},
    },
    changelog: true,
    pullRequest: {
      branchTemplate: "pubm/release/{scopeSlug}",
      titleTemplate: "chore(release): {scope} {version}",
      label: "pubm:release-pr",
      bumpLabels: {
        patch: "release:patch",
        minor: "release:minor",
        major: "release:major",
        prerelease: "release:prerelease",
      },
      unversionedChanges: "warn",
    },
  },
  ignore: [],
  tag: "latest",
  contents: ".",
  saveToken: true,
  releaseDraft: true,
  releaseNotes: true,
  rollback: {
    strategy: "individual",
    dangerouslyAllowUnpublish: false,
  },
  packages: [{ path: "." }],
  validate: {
    cleanInstall: true,
    entryPoints: true,
    extraneousFiles: true,
  },
  snapshotTemplate: "{tag}-{timestamp}",
  locale: "en",
});
```

## 레지스트리 추론 방식

`pubm`은 각 패키지 디렉터리의 매니페스트 파일을 검사해 레지스트리를 매핑합니다.

| 매니페스트 | 추론되는 레지스트리 |
|---|---|
| `package.json` | `npm` |
| `jsr.json` | `jsr` |
| `deno.json` / `deno.jsonc` | `jsr` |
| `Cargo.toml` | `crates` |

npm에는 추가 규칙이 두 개 있습니다.

- `package.json`의 `publishConfig.registry`가 있으면 `npm` 대신 해당 커스텀 URL을 사용합니다.
- `publishConfig`가 없을 때는 `.npmrc` 레지스트리가 `npm`을 대체합니다.

## 최상위 옵션

### `release`

- Type: `ReleaseConfig`

Configures version planning, changeset discovery, conventional commit parsing, changelog generation, and Release PR automation.

#### `release.versioning`

Controls how versions are assigned in multi-package repositories.

| Key | Type | Default | Description |
| --- | --- | --- | --- |
| `mode` | `"independent" \| "fixed"` | `"independent"` | Version packages independently or as one release. |
| `fixed` | `string[][]` | `[]` | Package groups that must resolve to the same version. |
| `linked` | `string[][]` | `[]` | Package groups that share the highest bump type. |
| `updateInternalDependencies` | `"patch" \| "minor"` | `"patch"` | Minimum bump for internal dependency ranges. |

#### `release.changesets`

| Key | Type | Default | Description |
| --- | --- | --- | --- |
| `directory` | `string` | `.pubm/changesets` | Directory for pending changeset files. |

#### `release.commits`

Configures conventional commit parsing. Commit analysis is always available; use `types` to decide which commit types create semver bump recommendations.

| Key | Type | Default | Description |
| --- | --- | --- | --- |
| `format` | `string` | `"conventional"` | Commit parser format. |
| `types` | `Record<string, "patch" \| "minor" \| "major" \| false>` | `{}` | Overrides for commit type to semver bump mapping. |

#### `release.changelog`

- Type: `boolean`
- Default: `true`

Whether pubm should write changelog entries when materializing release versions.

#### `release.pullRequest`

Configuration for the GitHub release PR workflow installed by `pubm workflow install github`.

- `branchTemplate` defaults to `pubm/release/{scopeSlug}` so one pending release PR is maintained per package or group scope while its computed version changes.
- `titleTemplate` defaults to `chore(release): {scope} {version}`.
- Release PR dry-run validation always runs before merge.
- `grouping` inherits `release.versioning.mode` by default and can be overridden with `"fixed"` or `"independent"`.
- `fixed` and `linked` inherit `release.versioning.fixed` and `release.versioning.linked` by default.
- Leave `grouping`, `fixed`, and `linked` unset to inherit. Setting `fixed: []` or `linked: []` under `release.pullRequest` intentionally disables inherited groups for release PRs.
- `unversionedChanges` defaults to `"warn"`. Use `"ignore"` to suppress the warning or `"fail"` to fail Release PR automation when only unversioned changes are found.

### `branch`

- 타입: `string`
- 기본값: `"main"`

가드된 릴리스 실행 동안 `HEAD`가 일치해야 하는 브랜치입니다.

### `packages`

- 타입: `PackageConfig[]`
- 기본값: `[{ path: "." }]`

배포 가능한 패키지를 명시합니다. 레지스트리는 패키지별로 덮어쓰지 않는 한 매니페스트 파일에서 추론됩니다.

### `plugins`

- 타입: `PubmPlugin[]`
- 기본값: `[]`

훅, 명령, 레지스트리, 생태계를 추가할 수 있는 플러그인을 등록합니다.

### `ignore`

- 타입: `string[]`
- 기본값: `[]`

워크스페이스 탐지에서 제외할 패키지 이름 또는 경로입니다.

### `tag`

- 타입: `string`
- 기본값: `"latest"`

publish 실행의 기본 dist-tag입니다.

### `contents`

- 타입: `string`
- 기본값: `"."`

릴리스 파이프라인을 실행하기 전에 `chdir`할 디렉터리입니다.

### `saveToken`

- 타입: `boolean`
- 기본값: `true`

지원되는 레지스트리 토큰을 로컬 머신에서 재사용하도록 저장할지 여부입니다.

### `skipDryRun`

- Type: `boolean`
- Default: `false`

prepare 단계에서 dry-run 검증을 생략합니다. workspace crate의 내부 의존성 새 버전이 아직 crates.io에 배포되지 않은 경우 유용합니다. `--no-dry-run-validation` CLI 플래그로도 설정할 수 있습니다.

### `releaseDraft`

- 타입: `boolean`
- 기본값: `true`

publish 흐름이 GitHub Release를 만들지 여부입니다. 비활성화하면 release 단계는 완전히 건너뜁니다.

### `excludeRelease`

- 타입: `string[]`
- 기본값: `[]`

버전이 올라가고 배포는 되지만 git 태그 생성과 GitHub Release 초안에서는 제외할 패키지를 지정하는 glob 패턴입니다. `release.versioning.mode`이 `"independent"`일 때만 적용됩니다.

레지스트리에는 배포하지만 단독 릴리스로 노출되지 않아야 하는 내부 빌드 산출물(예: 플랫폼별 바이너리)에 사용합니다.

```ts
export default defineConfig({
  release: {
    versioning: {
      mode: "independent",
    },
  },
  excludeRelease: ["packages/cli/platforms/*"],
  packages: [
    { path: "packages/core" },
    { path: "packages/cli" },
    { path: "packages/cli/platforms/*" },
  ],
});
```

### `validate`

- 타입: `ValidateConfig`

이 필드들은 공개 스키마의 일부지만, 아직 모든 publish 경로에서 활성 토글로 동작하지는 않습니다. 런타임 스위치가 아니라 타입이 있는 설정 표면으로 보아야 합니다.

### `snapshotTemplate`

- 타입: `string`
- 기본값: `"{tag}-{timestamp}"`

`pubm --snapshot`이 임시 버전 문자열을 만들 때 사용하는 템플릿입니다.

`package.json`의 기본 버전은 항상 스냅샷 버전 앞에 자동으로 추가됩니다. 템플릿은 기본 버전 뒤에 오는 접미사만 제어합니다.

사용 가능한 템플릿 변수는 다음과 같습니다.

- `{tag}`: snapshot 태그, 기본값은 `snapshot`
- `{timestamp}`: `YYYYMMDDTHHmmss` 형식의 UTC 타임스탬프
- `{commit}`: 현재 git 커밋 SHA

### `registryQualifiedTags`

- **Type:** `boolean`
- **Default:** `false`

When set to `true`, git tags in independent versioning mode are prefixed with the registry name to avoid collisions when packages from different ecosystems share the same name.

```ts
export default defineConfig({
  registryQualifiedTags: true,
  packages: [
    { path: 'js', registries: ['npm'] },
    { path: 'crates/cli', registries: ['crates'] },
  ],
})
```

Without this option, both packages named `my-tool` would produce the tag `my-tool@1.0.0`. With it enabled, tags become `npm/my-tool@1.0.0` and `crates/my-tool@1.0.0`.

pubm automatically detects name collisions and prompts you to enable this option. In CI environments, the collision causes an error with instructions to set this config option.

### `locale`

- 타입: `"en" | "ko" | "zh-cn" | "fr" | "de" | "es"`
- 기본값: `"en"`

CLI 출력 언어입니다. `--locale` 플래그 또는 `PUBM_LOCALE` 환경 변수로도 설정할 수 있습니다. 적용 우선순위: `--locale` 플래그 → `PUBM_LOCALE` 환경 변수 → config → 시스템 로케일 → `"en"`.

### `rollback`

publish 실패 시 롤백 동작을 제어합니다.

| 키 | 타입 | 기본값 | 설명 |
|----|------|--------|------|
| `strategy` | `"individual"` \| `"all"` | `"individual"` | 롤백 단위입니다. 더 이상 사용되지 않는 최상위 `rollbackStrategy`를 대체합니다. |
| `dangerouslyAllowUnpublish` | `boolean` | `false` | 비-TTY(CI) 환경에서 롤백 중 레지스트리 unpublish/yank를 허용합니다. unpublish 후에도 npm은 버전 번호를 영구적으로 예약합니다. 재사용할 수 없습니다. |

```ts
export default defineConfig({
  rollback: {
    strategy: "individual",
    dangerouslyAllowUnpublish: true, // CI에서 활성화
  },
});
```

더 이상 사용되지 않는 최상위 `rollbackStrategy` 필드는 계속 동작하며 `rollback.strategy`로 매핑됩니다. 둘 다 설정된 경우 `rollback.strategy`가 우선합니다.

## config와 CLI 플래그의 상호작용

현재 런타임 동작은 다음과 같습니다.

- config는 `packages`와 `plugins`의 저장소 기본값을 제공합니다.
- release planning config such as `release.versioning`, `release.changesets`, and `release.commits` is used by version and release PR flows
- CLI 플래그는 실행별 publish 동작을 계속 제어합니다.
- 레지스트리는 매니페스트에서 추론되며, `packages[].registries`가 그 추론을 덮어씁니다.

특정 레지스트리로 한 번만 publish하려면:

```bash
pubm --registry npm --tag beta
```

## 멀티 패키지 설정

워크스페이스나 혼합 언어 저장소에서는 패키지를 명시적으로 선언합니다. 레지스트리는 각 패키지의 매니페스트 파일에서 추론됩니다.

```ts
import { defineConfig } from "@pubm/core";

export default defineConfig({
  release: {
    versioning: {
      mode: "independent",
    },
  },
  packages: [
    {
      path: "packages/core",
    },
    {
      path: "packages/cli",
      registries: ["npm"],
    },
    {
      path: "crates/pubm-rs",
    },
  ],
});
```

다음 경우에는 명시적 패키지 설정이 좋습니다.

- 자동 탐지된 워크스페이스 중 일부만 publish하고 싶을 때
- JavaScript와 Rust 패키지가 같은 저장소에 함께 있을 때
- 추론된 레지스트리를 덮어쓰거나 private registry를 추가해야 할 때

### `PackageConfig`

패키지 수준 항목의 구조는 다음과 같습니다.

```ts
interface PackageConfig {
  path: string;
  registries?: Array<"npm" | "jsr" | "crates" | PrivateRegistryConfig>;
  ecosystem?: "js" | "rust";
  buildCommand?: string;
  testCommand?: string;
}

interface PrivateRegistryConfig {
  url: string;
  token: { envVar: string };
}
```

`registries`는 추론이 충분하지 않거나, 기본 제공 레지스트리와 함께 private registry를 추가해야 할 때만 사용합니다.

```ts
packages: [
  {
    path: "packages/a",
    registries: [
      "npm",
      { url: "https://npm.internal.com", token: { envVar: "INTERNAL_NPM_TOKEN" } },
    ],
  },
];
```

## 버전 전략

### 독립 버전 관리

각 패키지는 자체 버전과 태그를 가집니다.

```ts
export default defineConfig({
  release: {
    versioning: {
      mode: "independent",
    },
  },
});
```

### 고정 버전 관리

영향받는 모든 패키지가 하나의 버전을 공유합니다.

```ts
export default defineConfig({
  release: {
    versioning: {
      mode: "fixed",
    },
  },
});
```

### fixed와 linked 그룹

대부분의 패키지는 독립적으로 유지하면서, 선택한 패키지만 함께 움직이게 할 수도 있습니다.

```ts
export default defineConfig({
  release: {
    versioning: {
      mode: "independent",
      fixed: [["@acme/core", "@acme/react"]],
      linked: [["@acme/cli", "@acme/config"]],
    },
  },
});
```

- `fixed`: 그룹 안의 모든 패키지가 정확히 같은 버전이 됩니다.
- `linked`: 가장 높은 bump type이 그룹 전체에 전파됩니다.

## 스냅샷 설정

스냅샷은 preview 릴리스나 브랜치별 검증에 적합합니다.

```ts
export default defineConfig({
  snapshotTemplate: "{tag}-{timestamp}",
});
```

기본 버전은 자동으로 앞에 추가되므로, `"{tag}-{timestamp}"` 템플릿은 다음과 같은 출력을 만들어 냅니다.

- `0.4.1-snapshot-20260312T102233`

## 플러그인

`plugins` 배열을 통해 플러그인을 추가합니다.

```ts
import { defineConfig } from "@pubm/core";
import { externalVersionSync } from "@pubm/plugin-external-version-sync";

export default defineConfig({
  plugins: [
    externalVersionSync({
      targets: [
        {
          file: "README.md",
          pattern: /pubm@([\\w.-]+)/,
        },
      ],
    }),
  ],
});
```

훅 순서, 명령 등록, 커스텀 레지스트리 지원은 [Plugins API Reference](/pubm/ko/reference/plugins/)에서 확인합니다.

## 예시 패턴

### 단순 라이브러리

패키지에 `package.json`과 `jsr.json` (또는 `deno.json` / `deno.jsonc`)이 있으면 레지스트리는 자동 추론됩니다. 설정 파일은 필요하지 않습니다. `pubm.config.ts`는 선택 사항입니다. 기본값을 덮어써야 할 때만 만듭니다. 예를 들면 커스텀 브랜치, 플러그인, release assets 등이 있습니다.

### 앱 모노레포

`package.json`만 있는 패키지는 `npm`으로 해석됩니다. 덮어쓰려는 경우가 아니라면 `registries`를 명시할 필요가 없습니다.

```ts
import { defineConfig } from "@pubm/core";

export default defineConfig({
  release: {
    versioning: {
      mode: "fixed",
    },
  },
  packages: [
    { path: "packages/web" },
    { path: "packages/server" },
  ],
});
```

### JS + Rust 혼합 워크스페이스

```ts
import { defineConfig } from "@pubm/core";

export default defineConfig({
  release: {
    versioning: {
      mode: "independent",
    },
  },
  packages: [
    { path: "packages/sdk" },
    { path: "crates/sdk-core" },
  ],
});
```

## Release assets

`releaseAssets`는 각 GitHub Release에 첨부할 바이너리 산출물을 선언합니다. 대부분의 경우 단순한 glob 문자열만으로 충분합니다.

```ts
import { defineConfig } from "@pubm/core";

export default defineConfig({
  releaseAssets: [
    "platforms/*/bin/mytool",
  ],
});
```

### `compress`

- 타입: `"tar.gz" | "zip" | "tar.xz" | "tar.zst" | false | Record<string, CompressFormat>`
- 기본값: OS-aware 자동 감지 (`windows`는 `zip`, 그 외는 `tar.gz`)

전역 기본 압축 형식입니다. `releaseAssets` 안의 그룹이나 파일 수준에서 덮어쓸 수 있습니다.

```ts
export default defineConfig({
  // Windows는 zip, 그 외는 tar.gz 사용
  compress: { windows: "zip" },

  releaseAssets: [
    "platforms/*/bin/mytool",
  ],
});
```

### `releaseAssets`

- 타입: `(string | ReleaseAssetGroupConfig)[]`
- 기본값: `undefined` (첨부할 자산 없음)

각 항목은 glob 문자열이거나, 파일·압축·이름을 명시적으로 제어하는 객체입니다.

```ts
releaseAssets: [
  // 문자열: glob, 완전 자동
  "platforms/*/bin/mytool",

  // 객체: 그룹별 설정을 명시
  {
    packagePath: "packages/cli",    // 모노레포: 이 패키지의 release에 연결
    files: [
      "platforms/*/bin/mytool",
      { path: "dist/*.dmg", compress: false, name: "MyTool-{version}-{os}" },
    ],
    compress: "tar.gz",             // 그룹 기본값
    name: "{name}-{platform}",      // 그룹 이름 템플릿
  },
]
```

압축 cascade 규칙, 이름 템플릿 변수, 사용 사례 예시는 [Release Assets](/pubm/ko/guides/release-assets/)에서 확인합니다.

## Release analysis

`pubm changesets version` analyzes pending changeset files and conventional commit messages together. Changeset recommendations win when a changeset and a commit both cover the same package. Non-conventional commits, ignored commit types, and unmatched package changes are treated as unversioned changes; they never create version bumps by themselves.

Use `release.changesets.directory` to move the changeset file directory. Use `release.commits.types` to control which conventional commit types create semver bump recommendations.

```ts
export default defineConfig({
  release: {
    changesets: {
      directory: ".pubm/changesets",
    },
    commits: {
      types: {
        // Add types that are ignored by default
        refactor: "patch",
        docs: "patch",
      },
    },
  },
});
```

The default mapping is:

| Commit type | Bump |
| --- | --- |
| `feat` | `minor` |
| `fix` | `patch` |
| `perf` | `patch` |

Commits with `BREAKING CHANGE` in the footer or a `!` suffix (e.g. `feat!:`) always produce a `major` bump regardless of the type mapping.

Commit types not listed in the mapping (e.g. `chore`, `docs`, `test`, `ci`, `style`, `refactor`) are ignored during bump calculation. You can override any type in `release.commits.types`; for example, `refactor: "patch"` would treat refactor commits as patch bumps.

When Release PR automation finds only unversioned changes, `release.pullRequest.unversionedChanges` controls the outcome. The default is `"warn"`, which reports the unversioned changes and opens no PR. `"ignore"` suppresses the warning. `"fail"` fails the automation.
