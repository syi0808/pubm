---

`pubm`은 changeset을 "다음에 무엇을 릴리스해야 하는지"를 남기는 영속적인 기록으로 사용합니다. 커밋 메시지나 브랜치 이름에서 버전을 추론하는 대신, 체크인된 `.pubm/changesets/` 파일을 읽어 버전 상승과 changelog 항목으로 바꿉니다.

## Changeset이 중요한 이유

- 각 pull request 안에서 검토 가능한 릴리스 계획을 남길 수 있습니다.
- 패키지 간 버전 상승 계산을 결정적으로 수행할 수 있습니다.
- changelog 내용을 릴리스 이후가 아니라 릴리스 전에 작성할 수 있습니다.

## 파일 위치

보류 중인 changeset은 다음 위치에 있습니다.

```text
.pubm/changesets/
```

대화형 설정 마법사를 실행해 이 디렉터리를 초기화합니다.

```bash
pubm init
```

설정 중에 `pubm init`은 changeset 워크플로를 활성화할지 묻습니다. 활성화를 선택하면 `.pubm/changesets/` 디렉터리가 생성되고, PR 강제를 위한 `.github/workflows/pubm-changeset-check.yml`이 생성되며, changeset 파일을 추적하되 다른 `.pubm/` 콘텐츠는 무시하도록 `.gitignore`가 업데이트됩니다.

## Changeset 만들기

```bash
pubm changesets add
```

모노레포에서는 `pubm`이 어떤 패키지가 영향을 받는지와 각 패키지가 어떤 bump type을 받아야 하는지도 묻습니다.

비대화형으로 사용하려면:

```bash
pubm changesets add \
  --packages @acme/core,@acme/react \
  --bump minor \
  --message "Add token-based auth to the public client."
```

## Changeset 파일의 내용

```md
---
"@acme/core": minor
"@acme/react": patch
---

Add token-based auth to the public client.
```

메타데이터는 버전 계산을 제어합니다. 본문은 changelog 요약이 됩니다.

## 보류 중인 변경 확인

changeset을 소비하기 전에 먼저 확인합니다.

```bash
pubm changesets status
pubm changesets status --verbose
pubm changesets version --dry-run
```

## Changeset이 소비되는 방식

`pubm`으로 일반 릴리스 파이프라인을 실행하면 보류 중인 changeset은 버전 단계에서 소비됩니다.

그 흐름은 다음과 같습니다.

1. 보류 중인 `.pubm/changesets/*.md` 파일을 읽습니다.
2. 결과 패키지 버전을 계산합니다.
3. 매니페스트 버전을 업데이트합니다.
4. changelog 항목을 생성합니다.
5. 소비된 changeset 파일을 삭제합니다.
6. 릴리스 커밋과 태그를 생성합니다.

버전 작업만 미리 수행하고 싶다면 다음을 실행할 수 있습니다.

```bash
pubm changesets version
```

이 명령은 편집용 또는 통제된 워크플로에 유용하지만, `pubm`이나 `pubm --phase publish` 전에 반드시 실행할 필요는 없습니다.

## Changelog 생성

아직 버전을 소비하지 않고 changelog 텍스트만 만들고 싶다면:

```bash
pubm changesets changelog --dry-run
pubm changesets changelog --version 1.8.0
```

## Conventional commit recommendations

`pubm changesets version` analyzes conventional commits alongside pending changeset files. If a package has no changeset recommendation, a releasable conventional commit can create the bump recommendation. Non-conventional commits, ignored commit types, and commits that cannot be mapped to a package are treated as unversioned changes; they do not create bumps by themselves.

### Default commit type mapping

| Commit type | Bump |
| --- | --- |
| `feat` | `minor` |
| `fix` | `patch` |
| `perf` | `patch` |

A `BREAKING CHANGE` footer or a `!` suffix (e.g. `feat!:`) always produces a `major` bump.

### Configuring custom mappings

Override the default mapping in `pubm.config.ts`:

```ts
export default defineConfig({
  release: {
    commits: {
      types: {
        feat: "minor",
        fix: "patch",
        perf: "patch",
        docs: "patch",
      },
    },
  },
});
```

Release PR automation warns by default when only unversioned changes are found and no versioned release PR can be opened. Configure that with `release.pullRequest.unversionedChanges`: `"warn"` (default), `"ignore"`, or `"fail"`.

## `.changeset/`에서 마이그레이션

```bash
pubm changesets migrate
```

이 명령은 보류 중인 파일을 `.changeset/`에서 `.pubm/`으로 옮깁니다.
