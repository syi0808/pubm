---
title: Changesets
description: 릴리스 의도를 기록하고, changelog를 만들고, 버전 상승을 제어합니다.
---

`pubm`은 다음에 무엇을 릴리스해야 하는지 `.pubm/changesets/`의 파일로 기록합니다. 커밋 메시지나 브랜치 이름을 추론하지 않고, 체크인된 changeset 파일을 기준으로 버전과 changelog를 계산합니다.

## 왜 중요한가

- PR 안에서 검토 가능한 릴리스 계획을 남길 수 있습니다.
- 패키지별 버전 계산이 결정적입니다.
- changelog 문구를 릴리스 당일이 아니라 미리 작성할 수 있습니다.

## 위치와 초기화

```text
.pubm/changesets/
```

```bash
pubm init
pubm init --changesets
```

## changeset 만들기

```bash
pubm changesets add
```

모노레포에서는 어떤 패키지가 영향을 받는지와 각 패키지의 bump 타입도 함께 묻습니다.

비대화형 예시:

```bash
pubm changesets add \
  --packages @acme/core,@acme/react \
  --bump minor \
  --message "공개 클라이언트에 토큰 인증 추가"
```

## 파일 형식

```md
---
"@acme/core": minor
"@acme/react": patch
---

공개 클라이언트에 토큰 인증 추가
```

frontmatter는 버전 계산에 쓰이고, 본문은 changelog 요약이 됩니다.

## 대기 중 changeset 확인

```bash
pubm changesets status
pubm changesets status --verbose
pubm changesets version --dry-run
```

## changeset이 소비되는 방식

일반 릴리스 파이프라인에서 `pubm`을 실행하면 pending changeset은 버전 단계에서 자동으로 소비됩니다.

그 과정에서 `pubm`은:

1. `.pubm/changesets/*.md`를 읽고
2. 최종 패키지 버전을 계산하고
3. 매니페스트 버전을 갱신하고
4. changelog를 생성하고
5. 소비한 changeset 파일을 지우고
6. 릴리스 커밋과 태그를 만듭니다

버전 반영만 먼저 따로 하고 싶다면 다음 명령을 사용할 수 있습니다.

```bash
pubm changesets version
```

하지만 일반적인 `pubm` 또는 `pubm --ci` 흐름 전에 반드시 실행해야 하는 것은 아닙니다.

## changelog만 생성

```bash
pubm changesets changelog --dry-run
pubm changesets changelog --version 1.8.0
```

## `.changeset/`에서 마이그레이션

```bash
pubm changesets migrate
```
